2.9

CPU + IO Burst Scheduling — Priority Preemptive

⚙ CPU+IO Burst — Key Concept
Process Life CycleCPU Burst → IO Burst → CPU Burst → ... A real process alternates between CPU and IO device usage
Priority RuleLower number = Higher priority (stated in question)
ModePreemptive Priority — run 1 unit at a time, check priority at every step
IO DeviceEach process uses its IO device independently (parallel with CPU activity)
CPU IdleCPU goes idle when all processes are in IO — count this for utilisation
Completion TimeWhen a process finishes its LAST CPU burst — not after IO

2.10

Multi-Level Queue (MLQ) Scheduling

📥 Multi-Level Queue — Core Idea
Problem SolvedReal-life processes are of different types; a single queue is insufficient
SolutionSeparate queue for each process type, each with its own algorithm
Key DrawbackStarvation — lower-priority queues starve when high-priority queues always have work
MovementProcesses are permanently assigned to a queue; no movement between queues
▶ DIAGRAM 2.10-A — Multi-Level Queue Structure
① System Processes — Queue 1 (Highest Priority)
Algorithm: Round Robin • Examples: OS interrupts, kernel tasks
↓ (serves only when Queue 1 is empty)
② Interactive Processes — Queue 2 (Medium Priority)
Algorithm: SJF or any • Examples: VLC, browser, foreground apps
↓ (serves only when Queues 1 & 2 are empty)
③ Batch Processes — Queue 3 (Lowest Priority)
Algorithm: FCFS or any • Examples: background jobs, batch scripts
⚠ Starvation Problem: If System Process queue always has work, Interactive and Batch processes NEVER get CPU. Queue 3 may wait indefinitely.
Each queue has its own algorithm. Queues are served strictly top-to-bottom by priority.
💡 Key Difference from Standard Scheduling: In MLQ, processes are classified on arrival and placed in the appropriate queue permanently. The queue itself has a fixed priority, and processes in lower queues only get CPU when all higher queues are completely empty.
2.11

Multi-Level Feedback Queue (MLFQ) Scheduling

🔄 MLFQ — Core Idea (Eliminates Starvation)
Problem SolvedStarvation in MLQ — lower-priority processes never get CPU
Solution: FeedbackLow-priority processes are promoted to higher queues over time (Aging)
MovementProcesses CAN move between queues (upward promotion)
High PrioritySystem processes still run in their own queue without promotion needed
Each Queue TQLower queues get larger time quantum (TQ doubles as you go down)
▶ DIAGRAM 2.11-A — MLFQ Queue Structure & Promotion
RQ1 — Highest Priority, TQ=2 New batch process enters here
↓ If not done in TQ=2 → promote to RQ2
RQ2 — TQ=4 Gets more time than RQ1
↓ If not done in TQ=4 → promote to RQ3
RQ3 — TQ=8 Even more time
↓ If not done in TQ=8 → promote to RQ4
RQ4 — FCFS (no time limit) Runs to completion
Queues have increasing time quantum. Processes are promoted when they exceed a queue's TQ, getting faster CPU access each time.
▶ DIAGRAM 2.11-B — Worked Example: P1 (Batch Process, BT=19)
RQ1 (TQ=2)
P1 arrives at RQ1. Runs for 2 units. Remaining = 19−2 = 17.
Not finished → Promoted to RQ2. New space created in RQ1 for fresh processes.
RQ2 (TQ=4)
P1 runs for 4 units. Remaining = 17−4 = 13.
Not finished → Promoted to RQ3. Getting CPU faster than waiting in low priority queue.
RQ3 (TQ=8)
P1 runs for 8 units. Remaining = 13−8 = 5.
Not finished → Promoted to RQ4.
RQ4 (FCFS)
P1 runs for remaining 5 units. Completes! Total execution = 2+4+8+5 = 19 ✅
Total BT=19 distributed across queues: 2+4+8+5. Each promotion gives P1 faster CPU access, preventing starvation.
▶ TABLE 2.11-C — MLQ vs MLFQ Comparison
Parameter 📥 MLQ 🔄 MLFQ
Process movementFixed queue, no movementPromoted to higher queues (aging)
StarvationYes — lower queues may starveNo — aging prevents starvation
FlexibilityLow — strict classificationHigh — adapts over time
High priority handlingAlways runs first, no feedbackSystem processes run normally, no promotion needed
ComplexitySimplerMore complex (tracks queue level)
Real OS useLess common (strict)Windows, Linux schedulers use MLFQ-like systems
MLFQ is the evolution of MLQ that solves starvation while keeping process type separation.
⭐ Exam Point: MLFQ is used in real OS like Windows and Unix. The key idea is aging (promoting waiting processes). MLQ typically has no numerical questions — questions are theory-based. MLFQ may have a worked-example type question like the P1(BT=19) example above.

2.10
Multi-Level Queue (MLQ)
Core ideaMultiple queues, one per process type. Each has its own scheduling algorithm.
Queue typesSystem Processes (RR) → Interactive (SJF) → Batch/Background (FCFS)
PriorityStrictly top-down. Lower queues only run when ALL higher queues are empty.
MovementNone — processes stay in their assigned queue permanently.
★ StarvationYES — if system processes keep coming, batch processes may never run.
★ Exam NoteMLQ questions are purely theory. No numerical problems typically exist.
2.11
Multi-Level Feedback Queue (MLFQ)
SolvesStarvation problem of MLQ via Aging (promotion)
Aging / PromotionLow-priority process moves to higher queue after waiting. Gets CPU sooner.
TQ patternRQ1 TQ=2, RQ2 TQ=4, RQ3 TQ=8, RQ4 FCFS (doubles at each level)
Example (BT=19)RQ1: 2u done, 17 left → RQ2: 4u done, 13 left → RQ3: 8u done, 5 left → RQ4: 5u (FCFS done)
★ StarvationNO — aging guarantees every process eventually gets promoted and executed.
★ High prioritySystem processes run normally; feedback/promotion only applies to lower-priority queues.
Real OS useWindows, Unix, Linux all use MLFQ-inspired scheduling internally.
