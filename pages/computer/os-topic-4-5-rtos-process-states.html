Topic 4 — Remaining OS Types

⭐ Exam priority note: The topics below (RTOS, Distributed, Clustered, Embedded) rarely appear as direct questions in GATE/UGC NET. Questions are mostly from Multi-programmed, Multi-tasking, and Batch OS. However, knowing these is essential for completeness and for university/college exams.
⚡

Real-Time Operating System (RTOS)

Time is a hard constraint — results must arrive within a strict deadline
⏱️ Real-time vs Non-real-time — The Core Difference
Non-Real-Time (Batch)

You submit a job and get output after an hour, a day, or more. Time does NOT matter — only the correctness of the result matters.

Example: Uploading a video to YouTube, running a data analysis report. There is no urgency about when the result arrives.

Real-Time

Results must arrive immediately — within a precise deadline. Time is the constraint. Any delay = failure of the system.

Example: YouTube Live streaming — if you say something now, your audience must hear it right now, not after an hour.

Real-time does not mean fast — it means on time, every time, within a predetermined deadline.
🔴 Hard Real-Time vs 🟡 Soft Real-Time OS

🔴 Hard Real-Time

Deadlines are absolute and non-negotiable. Missing a deadline is a catastrophic failure — not just degraded performance. The system must complete tasks within a guaranteed time, every single time.

🚀 Examples
  • 🎯 Missile guidance systems — if missile's update arrives even 1 ms late, it hits the wrong target. No recovery is possible.
  • ✈️ Flight simulation / Autopilot — even a small delay in control signals can cause disasters mid-flight.
  • ❤️ Medical devices — pacemakers, life-support systems must respond within microseconds.
  • 🏭 Industrial robots — robotic arms in manufacturing; a timing error causes product damage or injuries.

🟡 Soft Real-Time

Deadlines are important but not catastrophic if missed. Occasional delays cause degraded quality or minor inconvenience — but NOT a system failure or critical situation.

🎮 Examples
  • 🎮 Computer gaming — a small frame delay is annoying but doesn't cause a critical failure.
  • 📺 YouTube Live streaming — if sound is delayed by 5–10 seconds, viewers notice it but no critical situation occurs.
  • 📱 Video conferencing (Zoom, Meet) — slight audio delay is acceptable; the call continues.
  • 🎵 Audio/video playback — a brief stutter is unpleasant but not dangerous.
Key distinction: Hard RTOS → missing deadline = catastrophic failure. Soft RTOS → missing deadline = degraded quality, not failure.
🎯 One-line exam answer: Hard RTOS has strict, absolute deadlines (missile systems, medical devices). Soft RTOS has flexible deadlines where missing them causes quality degradation, not failure (gaming, live streaming).

Distributed Operating System

Multiple geographically separated machines connected over a network — each with own resources

Core Idea: The computing environment is distributed geographically across the world. Multiple independent machines (each with its own CPU, memory, disk, and OS kernel) are connected through a network. They appear to users as a single unified system, but physically they are spread out.

↑ Loosely Coupled — Each machine is INDEPENDENT
Loosely coupled means: each machine has its own kernel, its own memory, its own disk. They work independently. They just happen to be connected and can share tasks over the network.
Examples: Facebook, Google — distributed data centres across multiple continents. Your request may be served by any machine worldwide.
✅ Advantages
  • Availability — if one machine fails, others serve the request
  • Fault Tolerance — system continues running even with partial failure
  • Scalability — add more machines to increase capacity
  • Resource Sharing — files, data, computation shared across sites
  • Geographically closer servers → faster response for local users
❌ Challenges
  • Network failures can disconnect components
  • Complex to manage — distributed coordination is hard
  • Security risks across multiple network locations
  • Consistency challenges — keeping all machines in sync

Clustered Operating System

Multiple machines connected via LOCAL network — work together as one super-computer

Core Idea: Multiple machines are connected through a Local Area Network (LAN) and work together as a single powerful unit. Users see it as one machine, but underneath, many machines are sharing the workload — like one giant supercomputer.

🔗 Clustered OS Architecture — Single Location, Multiple Machines
LAN — ONE Location (e.g., Data Centre)

Combined CPU power of all 4 machines = 4× processing power
Adding 100 machines in one cluster = computation power of 100 machines combined. Used in HPC (High-Performance Computing).

Common advantages of both: Availability, Fault Tolerance, Scalability, Load Balancing.
✅ Advantages (Both Distributed & Clustered)
  • Availability — if one machine fails, rest continue
  • Fault Tolerance — no single point of complete failure
  • Scalability — today X power, tomorrow 2X — just add more machines
  • Load Balancing — work distributed evenly across all nodes
  • Increased processing power (especially in clustered)
❌ Challenges
  • Higher cost (multiple machines + networking)
  • Software complexity increases significantly
  • Synchronisation between nodes is complex
  • Requires specialised networking infrastructure

Embedded Operating System

Fixed functionality — designed for a specific hardware device, cannot be modified

Core Idea: An embedded OS is built into a device for one specific purpose. The OS and application code are fixed and cannot be changed by the user. The device performs only the tasks it was originally designed for — nothing more.

📟 Embedded OS vs General-Purpose OS
📟 Embedded OS (Fixed)
  • 🔒 Works on ONE fixed set of functions only
  • 🔒 Cannot be reprogrammed by the user
  • 🔒 Hardware and software designed together for one purpose
  • 🔒 Washing machine: cannot cook food. Microwave: cannot wash clothes.
  • 🔒 OS is typically stored in ROM (Read-Only Memory)
Examples: Microwave oven, Washing machine, Air Conditioner, ATM machine, Elevator controller, Smart TV firmware, Car ECU
💻 General-Purpose OS (Flexible)
  • ✅ Can run any application the user installs
  • ✅ User can install new programs, change settings, modify code
  • ✅ Hardware is general — can run many different software combinations
  • ✅ Based on hard-wire or microprogramming — instructions can be adjusted
Examples: Windows, macOS, Linux on your laptop/desktop/phone
Embedded OS ≠ General-Purpose OS. Embedded = purpose-built, fixed. General = flexible, user-modifiable.
📌 Exam note: If asked "What is embedded OS?" — answer: An OS designed for a specific hardware device with fixed functionality that cannot be changed by the user. The OS is built into the device's ROM. Examples: microwave, washing machine, AC remote controllers.

Topic 5 — Process States & Life Cycle

💡 Important note (from lecture): The process state model is not a kernel-level technical mechanism — it is a conceptual model created to help us understand and explain how a process transitions from start to end. The kernel uses more complex internal structures, but this model is the standard way to describe process behaviour in all OS exams.
5A

Program vs Process

📄 Program
  • 📁 A static entity — just code stored on disk
  • 📁 Exists in secondary memory (hard disk)
  • 📁 Not executing — passively sitting in storage
  • 📁 Example: Firefox.exe saved on your C: drive
⚙️ Process
  • ⚡ A dynamic entity — a program in execution
  • ⚡ Loaded into RAM and given CPU time
  • ⚡ Actively running, consuming resources
  • ⚡ Example: Firefox you opened and are using right now
5B

The 5 Primary Process States

🆕 NEW
Process has just been created. The program was compiled and saved on secondary memory (hard disk). It hasn't been loaded into RAM yet.

Example: You wrote a C program and saved the .c file. Or you clicked to open Firefox — a new process is created.
✅ READY
Process has been loaded into RAM and is waiting in the Ready Queue for the CPU. It has all resources except the CPU itself.

The Long-Term Scheduler brings processes from secondary memory into RAM (ready state).
⚙️ RUNNING
Process is currently executing on the CPU. Instructions are being processed one by one.

The Short-Term Scheduler dispatches a process from Ready → Running. Only ONE process in running state in a uniprocessor system.
⏳ WAIT / BLOCK
Process is waiting for an I/O operation to complete (e.g., reading a file, accessing hardware). CPU is freed for another process.

Example: Process says "I need to read data from disk" → goes to Wait state → CPU serves another process.
🏁 TERMINATED
Process has completed all its instructions. The OS deallocates all resources (RAM, CPU time, file handles) that were assigned to it. The process no longer exists. Limited RAM must be freed up for new processes.

5D

All Transitions — What Causes Each One

🔄 Every State Transition Explained
TransitionTrigger / ReasonWho Does It?Type
NEW → READY Process is admitted into RAM (ready queue). OS decides to load it for execution. Long-Term Scheduler Normal
READY → RUNNING CPU is free; scheduler picks a process from ready queue and dispatches it to CPU. Short-Term Scheduler (Dispatch) Normal
RUNNING → TERMINATED Process completes all its instructions. All resources (RAM, CPU) are deallocated. OS (Deallocation) Normal
RUNNING → READY Pre-emption: (1) Higher priority process arrives in ready queue, OR (2) Time quantum expires (Round Robin). CPU is forcibly taken away. Short-Term Scheduler Preemptive
RUNNING → WAIT/BLOCK Process itself voluntarily requests I/O — it needs to read a file, write to disk, or access hardware. CPU is freed for another process. Process (self-initiated) Voluntary
WAIT → READY I/O operation completes. Process is moved back to ready queue. It does NOT go directly to Running — it must wait for CPU scheduling. OS (I/O completion interrupt) Normal
The RUNNING → READY transition is what makes a system pre-emptive. If this transition doesn't exist, the system is non-preemptive.
⚡ Preemptive vs Non-Preemptive Scheduling
⚡ Preemptive

The OS CAN forcibly remove a process from the CPU at any time — when a higher priority process arrives OR when the time quantum expires.

Running → Ready transition PRESENT
Multi-tasking OS, Round Robin = preemptive
🔒 Non-Preemptive

The OS CANNOT forcibly remove a process. Process runs until it finishes OR voluntarily gives up CPU (for I/O). No time quantum, no priority preemption.

Running → Ready transition ABSENT
Multi-programmed OS = non-preemptive
★ EXAM TIP: If the Running → Ready arrow exists in the state diagram → PREEMPTIVE. If it doesn't → NON-PREEMPTIVE.
5E

The 3 Schedulers — Roles & Differences

LT

Long-Term Scheduler (Job Scheduler)

NEW → READY · Controls degree of Multi-programming

Selects which processes from secondary memory (hard disk) should be brought into the Ready Queue (RAM). It controls how many processes are in memory at a time — this is the degree of multi-programming.

  • Runs infrequently (seconds to minutes between decisions)
  • Goal: bring more and more processes into the ready queue (fill the ready queue)
  • RAM is limited — cannot bring ALL processes in at once. LTS decides wisely.
  • Multi-programming concept = keeping the ready queue as full as possible
  • Controls the transition: NEW → READY
ST

Short-Term Scheduler (CPU Scheduler / Dispatcher)

READY → RUNNING · Runs most frequently

Picks a process from the Ready Queue and dispatches it to the CPU. This is the most active scheduler — it runs on every CPU context switch (every few milliseconds).

  • Runs very frequently — every time CPU needs a new process (milliseconds)
  • Uses CPU scheduling algorithms: FCFS, SJF, Round Robin, Priority Scheduling, etc.
  • Controls the transition: READY → RUNNING
  • Also handles RUNNING → READY in preemptive mode (time quantum / priority preemption)
  • Must be very fast — any overhead here wastes precious CPU time
MT

Medium-Term Scheduler (Swapper)

RAM ↔ Secondary Memory · Handles Suspend States

Temporarily removes processes from RAM and puts them in secondary memory (Swap Out) when RAM is full — and brings them back later (Swap In). This creates space for more important processes.

  • Handles Suspend Wait: When wait queue fills up → swap process out to secondary memory
  • Handles Suspend Ready: When ready queue fills up AND a high-priority process arrives → swap out a process to make space
  • Decides which process to swap out based on priority, LFU (Least Frequently Used), MFU, or other policies
  • Controls the Swap Out (RAM → Disk) and Swap In (Disk → RAM) transitions
  • Runs moderately frequently — between LT (slow) and ST (very fast)
5F

In the worst case, both the Wait Queue and the Ready Queue can fill up. Since RAM is limited, the OS must temporarily move some processes out to secondary memory. These are the Suspend (Additional) States.

🛒 Supermarket Analogy for I/O Wait State

You're in a supermarket queue for billing. Your bill is being processed (you are Running). Suddenly you realise you forgot to pick up an item — you say "hold on, I'll go grab it." The billing person doesn't wait for you — they immediately call the next customer (next process). When you come back (I/O done), you don't jump straight to the front — you go to the back of the queue (Ready Queue) and wait for your turn again.

Real-Time OS — Hard vs Soft
RTOS definition
OS where time is the primary constraint. Results must arrive within a specific deadline — not just correct, but on time.
Real-time vs Non-real-time
Non-real-time: submit job, result arrives whenever (hours/days). Real-time: result must arrive immediately, within deadline.
Hard Real-Time
Absolute, strict deadlines. Missing deadline = catastrophic failure. Examples: Missile systems, flight autopilot, medical pacemakers, industrial robots.
Soft Real-Time
Flexible deadlines. Missing deadline = quality degradation, NOT failure. Examples: Gaming, YouTube Live streaming, video conferencing.
Key difference
Hard: missing deadline → catastrophe. Soft: missing deadline → annoyance/degradation.

Distributed OS
Machines spread geographically (different cities/countries), connected via WAN/Internet. Each machine has its OWN kernel, memory, disk (loosely coupled).

Machines at same location, connected via LAN. Work together as ONE supercomputer. Increases computation power.

Facebook, Google — data centres worldwide. Each machine independent but networked.
Clustered example
University HPC cluster, AWS EC2 cluster — 100 machines in one building = 100× power.

Availability, Fault Tolerance (if one fails, rest continue), Scalability (add more machines), Load Balancing.
Embedded OS
Fixed-function OS built into a specific device. Cannot be changed. Examples: microwave, AC, washing machine, ATM. Stored in ROM.

Suspend States & Medium-Term Scheduler
Suspend Wait (Block Suspend)
Wait queue in RAM is full → Medium-Term Scheduler swaps waiting process out to secondary memory. Process continues I/O from disk.
Suspend Ready (Ready Suspend)
Ready queue full + high-priority (VIP/kernel) process arrives → MTS swaps a ready process out to secondary memory to make space.
Medium-Term Scheduler
Manages swap in/out (RAM ↔ secondary memory). Decides which process to swap based on priority, LFU, MFU, or other policy.
Swap Out
RAM → Secondary Memory. Process removed from RAM to free up space.
Swap In (Resume)
Secondary Memory → RAM. Process brought back when space is available.
Backing Store
If Suspend Wait process finishes I/O but Wait queue is still full → goes to Suspend Ready → eventually to Ready Queue.
