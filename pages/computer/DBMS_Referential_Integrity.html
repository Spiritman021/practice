What is Referential Integrity?

Referential Integrity means that the data across related tables must always remain consistent and contradiction-free. Specifically, any value in a Foreign Key column must always point to a valid, existing Primary Key value in the referenced table — no dangling or orphan references.

🔗 Integrity = having the same, consistent value for a piece of data across the entire database. The moment related tables show different values for the same fact, integrity is broken.
📊 Integrity vs No Integrity — Real-World Examples
❌ No Integrity — Disconnected Databases
A mobile phone priced at ₹14,999 on Amazon, ₹15,499 on Flipkart, ₹16,000 in a showroom.
Each store has an independent database with its own margins — no shared source of truth.
Result: Same product, three different prices — no integrity across systems.
✓ With Integrity — Connected University DB
Student "Ram" (Roll No. 1) is in the CSC department.
The Placement dept sees CSC. The Library sees CSC. The Exam dept sees CSC.
All departments read from the same connected database — one consistent truth everywhere.
If the Placement dept sees "CSC" but Library sees "Electronics" for the same Roll No. — referential integrity is broken.
Integrity = one consistent value for every fact across the entire connected system. Referential Integrity specifically governs FK-PK consistency.

Operations on the Referenced Table (Student / PK Table)

Any change to the Student table (the one holding the Primary Key) has implications for the Course table that depends on it. Here's what happens for each operation:

👑 Referenced Table (Student) — INSERT / DELETE / UPDATE
INSERT
✓ No Violation
You can freely add new students to the Student table. Adding a new student (e.g. Roll No. 4 — Sara) doesn't affect any existing Course records — no course is currently pointing to Roll No. 4.
New records in the parent table never break existing FK references.
DELETE
⚠️ May Cause Violation
Deleting a student may or may not cause a violation depending on whether the Course table references that student.

Violation case: Deleting Roll No. 1 (Ram) while Course table has rows pointing to Roll No. 1 (DBMS, CN) → creates orphan records in Course table.
Safe case: Deleting a newly admitted student who hasn't been assigned to any course yet → no problem.
🛠️ Three Solutions for DELETE Violation:
ON DELETE CASCADE
Deleting the parent row automatically deletes all child rows that reference it.
e.g. Delete Roll No.1 → DBMS and CN rows in Course are also deleted automatically.
ON DELETE SET NULL
Deleting the parent row sets the FK values in child rows to NULL.
⚠️ Cannot use if FK is also a PK in the child table (PK cannot be NULL).
ON DELETE NO ACTION
Default behaviour. If the parent row is referenced anywhere, the deletion is blocked with an error.
Safest option — nothing is deleted until the reference is resolved manually.
UPDATE
⚠️ May Cause Violation
Changing the Primary Key value in the Student table may cause inconsistency.

Example: Changing Roll No. 2 → 20 in the Student table, but the Course table still holds Roll No. 2 in its FK column → the FK now points to a non-existent PK value.
Same three solutions apply: ON UPDATE CASCADE, ON UPDATE SET NULL, ON UPDATE NO ACTION.

Here is the SQL syntax to define ON DELETE and ON UPDATE actions alongside the Foreign Key:

CREATE TABLE Course (
  Course_ID    VARCHAR(10) PRIMARY KEY,
  Course_Name  VARCHAR(80),
  Roll_Number  INTEGER,
  CONSTRAINT FK_Course FOREIGN KEY (Roll_Number)
    REFERENCES Student(Roll_Number)
    ON DELETE CASCADE     -- auto-delete child rows
    ON UPDATE CASCADE     -- auto-update FK to new PK value
);

Operations on the Referencing Table (Course / FK Table)

Changes to the Course table (the one holding the Foreign Key) are also constrained — because every FK value it holds must match an existing PK in the Student table.

🔗 Referencing Table (Course) — INSERT / DELETE / UPDATE
INSERT
⚠️ May Cause Violation
Inserting a new course record may or may not violate referential integrity depending on the FK value used.

Violation case: Assigning a course to Roll No. 7 when Roll No. 7 doesn't exist in the Student table → insertion blocked. You can't enroll a student who hasn't been admitted.
Safe case: Inserting a course with Roll No. 2, which already exists in Student → works perfectly fine.
The FK value must already exist in the parent's PK column before the child row can be inserted.
DELETE
✓ No Violation
Deleting a row from the Course table (e.g. removing a student from a course) has zero impact on the Student table. The parent data remains completely intact — we're only removing a child reference.
Deleting from the child table never affects the parent table.
UPDATE
⚠️ May Cause Violation
Updating the FK column (e.g. changing Roll No. from 2 to 20) may cause a violation.

Violation case: Changing the FK value to 20, but Roll No. 20 doesn't exist in Student → update blocked.
Safe case: Changing FK from 2 to 3, where Roll No. 3 already exists in Student → works fine.
The new FK value must also exist in the parent PK column — same rule as INSERT.
