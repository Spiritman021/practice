DBMS • CHAPTER 31 • DEPENDENCY PRESERVATION

3NF vs BCNF
Dependency Preservation

3NF always ensures dependency-preserving decomposition, but BCNF does not always preserve all original dependencies, even though it remains lossless.

Lossless DecompositionDependency Preserving 3NF vs BCNFCounterexample
1

Two Rules During Decomposition

Whenever a table is decomposed, two things are checked: the decomposition should be lossless and the dependencies should be preserved.

Lossless Decomposition
After joining the decomposed tables, we should get the original table back without extra or missing rows.
Dependency Preserving
Original functional dependencies should still be enforceable from the decomposed tables.
Important fact: 3NF always ensures dependency preservation. BCNF decomposition is lossless, but it may fail to preserve dependencies.
2

Given Counterexample

Consider relation R(A, B, C, D) with functional dependencies:

AB → CDD → A
StepResult
RHS attributesA, C, D appear on RHS, so B must be part of every candidate key.
B+Only B, so B alone is not a candidate key.
AB+AB gives C and D. Candidate key = AB.
CB+Only C and B, so not a candidate key.
DB+D gives A; then AB gives C and D. Candidate key = DB.
Candidate keys are AB and DB. Prime attributes are A, B, D. Non-prime attribute is C.

6

Dependency Preservation Fails

After decomposition, we can preserve D → A in R1 and derive BD → C in R2. But the original dependency AB → CD is not preserved directly.

ClosureBefore decompositionAfter BCNF decomposition
AB+A, B, C, D using AB→CDOnly A, B using D→A and BD→C
ConclusionOriginal dependency AB→CD holds.AB→CD is not inferable from decomposed dependencies.
Therefore, BCNF does not always ensure dependency-preserving decomposition.
