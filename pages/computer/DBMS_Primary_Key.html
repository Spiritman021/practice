Importance & The Lock–Key Analogy

The concept of keys is one of the most foundational topics in DBMS — heavily tested in college exams, placement interviews, and competitive exams like GATE and IBPS SO. It is deeply connected to Normalization and SQL.

🔐 The Lock & Key Analogy
🔒
Lock = Student Record
One specific student among 50,000 in the university database
⟷
🔑
Key = Primary Key
One specific value that opens exactly that one record — and no other
=
✅
Exact Match
The right key opens the right lock — one-to-one, no ambiguity
Just as a specific key opens only its lock, a Primary Key value uniquely identifies only one row in the entire table.
📚 Keys connect directly to Normalization (decomposing tables without data loss), SQL queries (WHERE clause lookups), and Foreign Keys (referential integrity). Mastering keys is non-negotiable.
2

Why Normal Attributes Fail as Keys (Duplicacy)

Imagine a university with 50,000 students. You record attributes like Name, Age, Date of Birth, Father's Name, and Mother's Name. None of these can reliably identify a unique student — because duplicates are possible in every single one of them.

❌ Normal Attributes — Why They Can't Be Keys
Name — Fails
Two students can have identical first and last names — e.g. "Rahul Sharma" is very common.
In a country of 1.4 billion, name uniqueness cannot be guaranteed in any university.
Date of Birth — Fails
Multiple students can share the exact same birthday — birthday twins are common.
In a batch of 500, statistically many students share DOB (Birthday Paradox).
Father's / Mother's Name — Fails
Two siblings enrolled in the same college will have identical parent names.
Common parent names repeat across thousands of unrelated families.
✕ None of these attributes guarantee uniqueness across all rows — they cannot reliably identify a single student.
Regular attributes can have duplicate values — making them useless as row identifiers in a large dataset.
3

Candidate Keys — Unique But Not Enough

Attributes that do not contain duplicate values are called Candidate Keys. Their defining property is uniqueness. For a student, these include Phone Number, Aadhaar, PAN Card, Driving License, Registration Number, Roll Number, and Email ID.

But uniqueness alone is not sufficient. Candidate keys have a critical weakness — they can sometimes be empty (NULL) or contain shared/incorrect data.

⚠️ The NULL Problem — Why Candidate Keys Alone Aren't Enough
📱 Phone Number Issue
• Student might provide a wrong number or leave it empty.
• Two siblings in the same college might share the same parent's phone.
• Result: NULL or DUPLICATE → search returns wrong/no results.
🪪 Aadhaar / PAN Issue
• Not all 50,000 students in a university have an Aadhaar or PAN card yet.
• A first-year student aged 17 may not have applied for PAN.
• If column is NOT NULL, valid students are blocked. If allowed NULL: NULL → broken identity.
🔑 Candidate Key property = Unique only. It can still be NULL. That's the gap a Primary Key fills.
Candidate Keys are unique — but they can be null or sometimes practically shared. A Primary Key eliminates this weakness.
4

The Primary Key: Unique + Not Null

A Primary Key is a Candidate Key that satisfies two strict, non-negotiable properties simultaneously: it must be Unique AND Not Null. Out of all candidate keys, the designer picks exactly one — the most appropriate — as the Primary Key.

🔑 Primary Key — Two Mandatory Properties
🔢
UNIQUE
No two rows can ever have the same Primary Key value. Each row gets a one-of-a-kind identifier.
+
🚫
NOT NULL
The field can never be left empty. Every row must have a valid PK value — always.
=
⭐
PRIMARY KEY
The chosen candidate. The official, guaranteed row identifier. One per table — no exceptions.
UNIQUE ensures no confusion between rows. NOT NULL ensures every row always has an identity. Together = Primary Key.
📊 Why Aadhaar Fails as PK — But Roll Number Works
❌ Aadhaar as Primary Key — Fails
If Aadhaar is set as PK with NOT NULL constraint, and a student doesn't have one:

• College must either deny admission to the student, or
• Fill the column with garbage values like $, #, or XXXX — which defeats the purpose entirely.

Not all 50,000 students are guaranteed to have an Aadhaar. Inappropriate as PK.
✓ Roll/Registration Number — Perfect PK
• Automatically generated by the system at the time of admission.
• Always unique — the system ensures no two students get the same number.
• Always assigned — no student ever leaves without one. Never NULL.
• The student didn't choose it — the system gave it. Perfect Primary Key.
The best Primary Key is one the system generates and assigns — not one collected from the user.
5

Primary Keys Are Assigned, Not Taken

A golden rule of database design: we never take the Primary Key value from the user — the system gives it to them. You don't get to choose your roll number when joining college; the college assigns it automatically. This guarantees both uniqueness and non-nullness with zero chance of user error.

🌍 Real-World Examples — System-Assigned Primary Keys
🎓 University
🎓
Registration Number automatically generated at admission. Student never chooses it.
REG/2024/CS/001
🛂 Passport Office
🛂
Passport Number auto-generated when application is approved. You receive it — you don't pick it.
P1234567
🚗 License Office
🚗
Driving License Number automatically assigned after test. System ensures no duplicates across all states.
DL-09-2024-0012345
In every system, the Primary Key is generated and assigned by the system — never chosen by the user. This guarantees Unique + Not Null without any user involvement.
6

Candidate Key vs Primary Key — Side by Side

⚖️ Candidate Key vs Primary Key — Full Comparison
🗝️ Candidate Key
✅
Unique — values never repeat across rows.
⚠️
Can be NULL — field can be left empty.
📦
Multiple allowed — a table can have many candidate keys.
Examples: Phone, Aadhaar, PAN, Email, Voter ID…
⭐ Primary Key
✅
Unique — values never repeat across rows.
✅
NOT NULL — field can never be left empty. Always has a value.
☝️
Only ONE — a table can have exactly one Primary Key.
Best choice: Roll Number, Registration Number (system-assigned)
Primary Key = Candidate Key + NOT NULL constraint + Only one per table. It is the most important constraint in relational database design.
7

Golden Rules for Exams

⭐ Golden Rules — Exam-Ready Facts
1
Candidate Key property = Unique ONLY
A Candidate Key guarantees uniqueness but can still be NULL. It is not required to always have a value.
UNIQUE
2
Primary Key property = Unique + Not Null
A Primary Key must be both unique AND non-empty. Both conditions are mandatory — one without the other is not a Primary Key.
UNIQUE + NOT NULL
3
Multiple Candidate Keys — One Primary Key
A table can have many Candidate Keys (Phone, Aadhaar, PAN, Email…) but only ONE Primary Key. No DBMS allows more than one.
1 PK ONLY
4
System Assigns PK — User Never Chooses It
The Primary Key is generated by the system, not provided by the user. This guarantees it's always unique and never null.
AUTO-ASSIGNED
5
No DBMS Allows Two Primary Keys
Oracle, SQL Server, MySQL, IBM DB2 — none of them permit more than one Primary Key per table. If one attribute uniquely identifies rows, there's no reason for another.
ALL DBMS
These five rules appear directly in GATE, UGC NET, IBPS SO, and placement interviews — memorize them exactly.

Why Normal Attributes Fail as Keys
Problem:Normal attributes like Name, Age, DOB, Father's Name can be duplicated across different students.
Name:"Rahul Sharma" may appear hundreds of times in a 50,000-student database.
DOB:Multiple students can share the same birthday — statistically inevitable in large batches.
Parent names:Siblings enrolled together have identical parent names.

3
Candidate Key — Unique But Can Be Null
Definition:Attributes with no duplicate values — inherently unique. Phone, Aadhaar, PAN, Email, Driving License, Roll No.
Property:Unique only. Can still be NULL (empty).
Phone NULL issue:Student may enter wrong number, leave blank, or two siblings share one parent's phone.
Aadhaar/PAN issue:Not all 50,000 students have Aadhaar or PAN cards — especially younger students. Can't enforce NOT NULL.
Gap:Uniqueness alone ≠ perfect identifier. The NULL possibility breaks row identification.
4
Primary Key — Unique + Not Null
Definition:A Candidate Key with two mandatory properties: UNIQUE + NOT NULL. Chosen by the designer — one per table.
Not Null means:The field can never be empty. Every inserted row must provide a PK value.
Why Aadhaar fails as PK:Not all students have one. If forced NOT NULL → college must deny admission or fill garbage values.
Best PK for students:Registration Number (university) or Roll Number (college) — system-generated, always unique, never null.
5
Primary Keys Are Assigned, Not Taken
Golden rule:Never take the PK value from the user — the system generates and assigns it.
University:Registration Number auto-assigned at admission. Student doesn't choose it.
Passport Office:Passport Number auto-generated when approved.
License Office:License Number auto-generated after test clearance.
Why this works:System-generated → guaranteed unique (no collision) + guaranteed assigned (never null). User error eliminated.
