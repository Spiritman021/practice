Centralized and Client–Server Architecture

How DBMS work moves from one mainframe to clients, servers and multiple tiers.

← PreviousQuizNext →

1. Architecture Overview

DBMS architecture decides where data is stored, where it is processed, and how users access it.

A centralized DBMS puts almost everything on one central computer. Client–server architecture distributes work between user-facing clients and back-end servers.

2. Centralized DBMS Architecture

All database operations, applications and interfaces are controlled by one powerful central computer—usually a mainframe or server.

How it works

  1. Users enter commands through low-processing terminals.
  2. Terminals send commands to the mainframe.
  3. The mainframe executes them and accesses the database.
  4. Results return to terminals for display.

Early systems used this model because most processing power existed centrally.

University example: Campus terminals send grade and registration requests to one mainframe, which processes them and returns results.

BenefitsDrawbacks
Centralized control: one place controls access, making security simpler.Scalability issue: more users can overload the mainframe.
Simple maintenance: updates are applied once, reducing administration.Single point of failure: mainframe failure stops the complete system.

3. Client–Server DBMS

Client–server architecture evolved from centralized systems by dividing work:

Client

User-facing machine or application that accepts input, performs local work and displays results.

Server

Back-end system that stores the database and performs data processing.

The client sends a query; the server processes it and returns information. Work distribution balances resources, reduces central load, enables remote access and allows clients and servers to be on different machines.

Example: Customer-service desktops query a central customer database server.

4. Two-Tier Client–Server

The client directly communicates with the database server.

Client: UI + Application Logic⇄Server: Storage + Retrieval Direct connection

Bank example: Teller software requests an account balance from the database server and displays the returned value.

BenefitsDrawbacks
Simple: straightforward and easy to implement.Scalability limit: many clients can bottleneck the server.
Direct interaction: fast for simple tasks.Client maintenance: application changes may require updating every client.

5. Three-Tier Client–Server

Three-tier architecture adds an application server between client and database.

TierRole
Presentation Layer (Client)Displays data and collects user input.
Application Layer (Middle)Processes requests, business rules and database interaction.
Data Layer (Server)Stores and manages the database.

Restaurant example: Browser → application server processes the order → database server stores/retrieves order data.

  • Better load management: middle tier processes requests before the database.
  • Enhanced security: it validates users and acts as a gatekeeper.
  • Complex logic: business rules stay in a dedicated tier.

Healthcare example: The application server authenticates users before allowing controlled access to patient data.

6. Beyond Three Tiers: n-Tier

An n-tier system adds specialized layers such as authentication, data aggregation or application services. CRM and ERP systems commonly use this structure.

  • Modularity: teams develop and manage layers independently.
  • Resilience: a failure can be handled within one layer instead of stopping the entire system.
  • More layers also improve flexibility and scalability, though deployment becomes more complex.

7. Quick Comparison

ArchitectureStructureMain point
CentralizedTerminals → one mainframeEasy control; single failure point
Two-tierClient ↔ database serverSimple/direct; client maintenance and server bottleneck
Three-tierClient ↔ application server ↔ databaseBetter security, load distribution and business logic
n-tierMultiple specialized layersModularity, scalability and resilience

8. IBPS SO Quick Revision

  • Centralized terminals mainly perform input/output; mainframe performs processing.
  • Centralized advantage: control and simple maintenance.
  • Centralized weakness: poor scalability and single point of failure.
  • Client handles user interaction/local processing; server handles database work.
  • Two-tier has direct client–database-server communication.
  • Three-tier adds an application server for business logic and security.
  • Presentation → Application → Data is the three-tier order.
  • n-tier adds specialized services; major benefits are modularity and resilience.

Most likely questions: Central failure stops all—centralized. Direct client/database connection—two-tier. Gatekeeper/business logic—application tier. Independent specialized layers—n-tier.
