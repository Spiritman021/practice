Topic 8 — fork() System Call Basics

8A

What is fork()?

fork() is a process control system call that creates a child process — an exact duplicate of the calling (parent) process. The moment fork() executes, two processes exist: the original (parent) and the newly created (child). Both continue executing from the point after fork() — in parallel (concurrently).

⚡ Both P and C1 now run concurrently (in parallel). Parent was NOT killed — it is still alive.
The parent process is not replaced. It continues running. The child is a new, separate process that starts from the same point as the parent.
🌿 fork() — Full Clone
Creates a completely new process — own PID, own memory space, own registers, own file descriptors. Heavy — copying everything is expensive. Code section may be shared (copy-on-write), but all other resources are separate.
🧵 Thread — Lightweight
Like creating an extra "hand" — shares most of the parent's resources (memory, code, files). Only the stack and registers are separate. Much lighter than fork(). Used when you want parallel work with minimal overhead.
8B

Return Values of fork()

↩️ fork() Return Values — Three Possible Cases
0
Child Process
fork() returns 0 to the child. This is how the child knows it IS the child. Used in if-conditions to distinguish child from parent.
+N
Parent Process
fork() returns the child's PID (a positive number) to the parent. Parent can use this to track or wait for the child.
-1
Error — No Child
fork() failed — child was NOT created. Usually because kernel is too busy or system ran out of resources. Ignore this in exam questions unless explicitly mentioned.
For exam purposes: 0 = child, positive = parent. The -1 case is mentioned for completeness — exam questions will not use it unless they say so.

Topic 9 — fork() with Logical Operators (Questions)

When fork() is used inside an if condition with logical operators (&& or ||), we cannot directly apply the 2ⁿ formula. Instead, we must trace each process individually, applying short-circuit evaluation rules and fork() return values step by step.

9A

Short-Circuit Evaluation Rules

⚡ AND (&&) vs OR (||) — Short-Circuit Rules with fork()
&& (AND) Operator
Left valueRight evaluated?Result
0 (false)NO — short circuits0 (false)
+ve (true)YES — must check rightdepends on right
+ve && +ve—true → if executes
+ve && 0—false → if skips
Child (0) && anything → always 0 (false) — child never evaluates the second fork. Child skips the if-body and goes straight to printf.
|| (OR) Operator
Left valueRight evaluated?Result
0 (false)YES — must check rightdepends on right
+ve (true)NO — short circuitstrue → if executes
0 || 0—false → if skips
0 || +ve—true → if executes
Parent (+ve) || anything → always true (short-circuits) — parent skips the second fork. Parent goes directly into the if-body without calling fork() again.
These are standard C language evaluation rules — child=0=false, parent=positive=true. Apply exactly as in a normal logical expression.
9B

Solved: if( fork() && fork() ) — Complete Trace

📋 Program with if(fork() && fork())
main() { if( fork() && fork() ) { // if-body: reached only when BOTH fork() calls return non-zero } printf("Hello"); // ALL processes reach here }
🔍 Step-by-Step Trace — Who Creates Which Process, Who Prints
⚙️ The program starts as one process — call it Parent (P). We evaluate the if-condition left to right.
📌 FIRST fork() — left side of &&
C1 (child) fork() is called → C1 is created. C1 receives return value 0.
C1 evaluates: if( 0 && ... ) → 0 && anything = 0 = false.
Short-circuit kicks in: second fork() is NOT evaluated by C1. C1 skips the if-body.
C1 reaches printf → C1 prints "Hello" (1 time).
P (parent) fork() returns +PID (positive) to P.
P evaluates: if( +PID && ... ) → first operand is true. Must check second fork().
P continues to the second fork().
📌 SECOND fork() — right side of && (only P evaluates this)
C2 (child of P) P executes second fork() → C2 is created. C2 receives return value 0.
P now has: if( +PID && 0 ) → positive && 0 = false.
Both P and C2 skip the if-body.
C2 reaches printf → C2 prints "Hello". P also reaches printf → P prints "Hello".
⚠️ Wait — does the if-body execute for anyone?
The if-body only executes when both fork() calls return positive (both true). Let's check:
• C1: 0 && (skipped) → false → ❌ no
• C2: (positive && 0) → false → ❌ no
• P: (positive && 0) → false → ❌ no
Nobody enters the if-body in this program because whenever the second fork() runs, it returns 0 to the new child (C2) making it false, and P also gets 0 from second fork — so result is false for P too.
✅ Final Answer
"Hello" printed by: C1 + C2 + P = 4 times
Child processes created: C1 (from fork#1) + C2 (from fork#2) = 2
Wait — we said 4 times but counted C1+C2+P = 3. Let's recheck…
Clarification (important!): The lecture says 4 times. Let's trace again carefully:

When P evaluates if(fork() && fork()):
• fork#1 creates C1. Now P and C1 both execute from here.
• C1 sees 0 → short-circuits → skips fork#2 → prints "Hello". (C1 = 1 print)
• P sees +PID → evaluates fork#2 → fork#2 creates C2. C2 sees 0, P sees +PID.
• C2: +PID && 0 = false → if skips → C2 prints "Hello". (C2 = 1 print)
• P: +PID && +PID → true → P enters if-body (the if-body itself has no fork, or in this version just falls through) → P prints "Hello". (P = 1 print)

Wait — C2 got 0 but P got a positive from fork#2. So P's condition = +PID && positive = TRUE!
That means P enters the if-body. But there's no fork inside the if-body, so P goes to printf → P prints Hello.
Total = C1 + C2 + P = 3 prints. The lecture answer of 4 accounts for a fork inside the if-body in the full original question shown. Let's trace the lecture's exact code below.
The trace depends on the exact code. The key skill is: trace each process separately, apply return values (0 for child, positive for parent), apply short-circuit rules.

9D

Practice: if( fork() || fork() ) — Trace

📋 if(fork() || fork()) — How Behaviour Changes with OR
main() { if( fork() || fork() ) { // if-body: reached when ANY fork() returns positive } printf("Hello"); }
P (parent) — fork#1 returns +PID:
if(+PID || ...) → short-circuit! +PID is true → OR is already satisfied. fork#2 is NOT called by P.
P enters if-body → then reaches printf → P prints "Hello" ✅
C1 (child of fork#1) — fork#1 returns 0:
if(0 || ...) → 0 is false → must evaluate right side → C1 calls fork#2.
fork#2 creates C2. C2 returns 0 to itself, +PID to C1.
C1 sees: if(0 || +PID) → true → C1 enters if-body → C1 prints "Hello" ✅
C2 sees: if(0 || 0) → false → C2 skips if-body → C2 prints "Hello" ✅
Total "Hello" prints = 3 (P + C1 + C2)  |  2 child processes (C1, C2)
P and C1 enter if-body. C2 doesn't. But ALL reach printf — printf is outside if, so all 3 print "Hello".
With OR: Parent short-circuits (skips fork#2). Child must evaluate fork#2. Exactly the opposite of AND behaviour.
⚡ AND vs OR — Key Behaviour Summary
Conditionif(fork() && fork())if(fork() || fork())
Child (0) → left side 0 && ? → false, fork#2 NOT called, child skips if-body 0 || ? → must check right, child CALLS fork#2
Parent (+) → left side + && ? → must check right, parent CALLS fork#2 + || ? → already true, fork#2 NOT called, parent enters if-body
New child of fork#2 Returns 0 to new child → + && 0 = false → if skips Returns 0 → 0 || 0 = false → if skips
Total Hello prints 4 (with a fork in if-body) or 3 (no if-body fork) 3 (P + C1 + C2)
Who enters if-body Only P (positive && positive = true) P and C1 (both evaluate to true via short-circuit or || +PID)
The golden rule: AND = child short-circuits (fork#2 saved). OR = parent short-circuits (fork#2 saved). In both cases the "new child of fork#2" never enters the if-body.

fork() — Core Concepts
What fork() does
Creates a child process — an exact clone of the calling (parent) process. Both run concurrently from the point after fork().
Parent NOT killed
fork() does NOT replace the parent. The parent continues running. You now have TWO processes running in parallel.
fork() returns 0
Returns 0 to the child process. The child uses this to know it is the child.
fork() returns +PID
Returns the child's PID (positive number) to the parent. Parent uses this to track/wait for child.
fork() returns -1
Error — child NOT created (kernel too busy). Ignore in exam questions unless explicitly mentioned.
fork() vs thread
fork() = full clone (own PID, memory, registers) — heavy. Thread = shares most resources, just extra execution unit — lightweight.
2
The Two Formulas — MUST Memorise
Total processes / printf runs
2ⁿ — where n = number of unconditional fork() calls. Includes parent + all children.
Total child processes
2ⁿ − 1 — total minus 1 (the original parent is not a child).
n=1 → 2¹=2
2 total processes, 1 child. printf prints 2 times.
n=2 → 2²=4
4 total processes, 3 children. printf prints 4 times.
n=3 → 2³=8
8 total processes, 7 children. printf prints 8 times.
⚠️ Limitation
These formulas ONLY work for unconditional fork() calls (no if/while with &&/|| operators). Use step-by-step trace for conditional forks.
3
fork() with && (AND) Operator
Child (returns 0)
0 && anything = 0 (false). Short-circuits — second fork() NOT called by child. Child skips if-body, goes to printf.
Parent (returns +PID)
+PID && ? → must check second operand. Parent evaluates the second fork().
Result of + && 0
False → process (P or new child C2) skips if-body, goes to printf directly.
Result of + && +
True → process enters if-body (only parent P reaches this case).
if(fork() && fork()) answer
Processes: P, C1, C2. C1 short-circuits (doesn't call fork#2). P calls fork#2 → C2. Total Hello = 3 (or 4 if fork in if-body creates C3).
4
fork() with || (OR) Operator
Parent (returns +PID)
+PID || anything = true. Short-circuits — second fork() NOT called by parent. Parent enters if-body directly.
Child (returns 0)
0 || ? → must check second operand. Child evaluates the second fork().
0 || 0 (new child C2)
False → C2 skips if-body, goes to printf.
0 || + (C1)
True → C1 enters if-body.
if(fork() || fork()) answer
Processes: P, C1, C2. P short-circuits (no fork#2). C1 calls fork#2 → C2. P and C1 enter if-body. C2 doesn't. Total Hello = 3.
Golden rule
AND = child short-circuits. OR = parent short-circuits. Opposite behaviours!
5
How to Solve Any fork() Question
Step 1
Identify if fork() calls are unconditional (plain, no if/&&/||) → use 2ⁿ formula directly.
Step 2
If fork() is inside if conditions with &&/|| → trace step-by-step. Do NOT use 2ⁿ directly.
Step 3 (trace)
For each process, note what fork() returned. Apply the logical operator rules. Determine if if-body executes.
Step 4
Remember: ALL processes (whether they entered if-body or not) will execute the code AFTER the if-block (like printf). Count all of them.
Key fact
0 = false = child. Positive = true = parent. These are standard C truthiness rules applied to fork() return values.
