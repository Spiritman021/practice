DBMS Data Schemas

Schema types and database instances—short and exam-focused.

← PreviousQuizNext →

1. Database Schema

A database schema is the database’s skeleton or blueprint. It represents the logical view of the entire database.

It defines:

  • how data is organized;
  • entities and relationships among them;
  • tables and views; and
  • constraints that data must follow.

A schema contains descriptive details, not actual records. A schema diagram can visually show this structure.

Database designers create the schema so programmers can understand the database and build useful applications.

Easy analogy: Schema is a building plan—it shows rooms and rules, but it does not contain the people living inside.

2. Types of Database Schema

Physical Database Schema

Defines how and where data is actually stored in secondary storage.

  • Files
  • Indexes
  • Storage formats and locations

Logical Database Schema

Defines the database’s logical organization and rules.

  • Tables
  • Views
  • Relationships
  • Integrity constraints
SchemaMain questionIncludes
PhysicalHow is data stored?Files, indexes, secondary-storage details
LogicalWhat structure and rules exist?Tables, views, relationships, constraints

Exam clue: “Files, indexes or secondary storage” → Physical schema. “Tables, views or integrity constraints” → Logical schema.

3. Database Instance

A database instance is the state of an operational database at a particular moment. It contains the actual data—a snapshot of the database at that time.

  • The instance changes when rows are inserted, updated or deleted.
  • Different times can have different instances under the same schema.
  • The DBMS checks validations, constraints and conditions created by designers.
  • Therefore, every permitted instance should remain in a valid state.

Easy analogy: Schema = empty class register format. Instance = names and attendance written in it today.

Do not confuse: Creating, deleting or updating a row changes the instance. Adding a column or changing a constraint changes the schema.

5. IBPS SO Quick Revision

  • Schema = skeleton/logical view of the complete database.
  • It defines organization, relationships and constraints.
  • Schema diagrams visually represent descriptive database details.
  • Database designers create schemas for programmers and applications.
  • Physical schema defines files, indexes and secondary storage.
  • Logical schema defines tables, views and integrity constraints.
  • Schema contains no actual data.
  • Instance = actual database snapshot at a given time.
  • Instances change frequently; schemas change rarely.
  • DBMS uses constraints and validations to keep each instance valid.

Most likely questions: Database snapshot—instance. Database skeleton—schema. Files and indices—physical schema. Tables and integrity constraints—logical schema.
