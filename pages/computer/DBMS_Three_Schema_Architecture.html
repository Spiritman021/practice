Three Schema
Architecture

Three levels of abstraction — External, Conceptual, and Physical — and how they achieve data independence for users, designers, and DBAs.

GATEUGC NET IBPS SO ITB.Tech / BCA
1

What is a Schema? (Quick Recap)

A schema is the structure used to store data. In a relational database, this means defining 2-dimensional tables made of rows and columns. It also specifies the characteristics of each column — for example, declaring a "roll number" as an INTEGER of 2-byte or 4-byte size.

📌 Schema = Structure of the database. It defines what tables exist, what columns they have, and what data types those columns use. The actual data rows come later — the schema is just the skeleton.
2

The Main Motive: Data Independence & Abstraction

The Three-Schema Architecture was introduced in 1970 with one primary goal: achieving data independence and data abstraction. These two concepts are the core reason this architecture exists.

🙈 Data Abstraction
Hiding physical storage details from the user.

When you use Gmail, your emails are physically stored somewhere — Delhi, USA, some data center — in some specific file format. But Google hides all of that from you. You just see your inbox. That hiding is data abstraction.
🔗 Data Independence
Users and physical data should not interact directly.

The three layers sit in between — so if the physical storage changes (new server, new file format), the user experience stays exactly the same. Users access data through the abstraction layers, not the raw files.
3

The Three Levels of Abstraction

🏛️ Three-Schema Architecture — Complete Tower
LEVEL 3 — EXTERNAL External Schema  /  View Level
Purpose: Defines how data is shown to the user. Each user or user group gets a customized view — only the data relevant to their role is visible. Enforces authorization (read-only vs read-write).

Examples: Student portal (marks, fees), Faculty portal (enter marks, apply leave), Admin portal (all records).

Technology: Built using PHP, Java, HTML, front-end frameworks.
👨‍💻 Front-end / Interface Designers
⇕
LEVEL 2 — CONCEPTUAL ⭐ Conceptual Schema  /  Logical Level
Purpose: Defines the complete logical blueprint of the database — all tables, columns, relationships, and constraints. Like an architectural blueprint before building a house.

Defines: Which tables to keep, column names and data types, how tables relate to each other (e.g. Student ↔ Course via Enrollment).

Model used: The E-R (Entity-Relationship) Model is the primary tool at this level.
🗂️ Database Designers
⇕
LEVEL 1 — INTERNAL Physical Schema  /  Internal Level
Purpose: Decides where and how data is physically stored on disk. This layer is completely hidden from users and designers.

Defines: Actual hard disks, file names, file sizes, fragmentation, whether storage is centralized (one location) or distributed (multiple global servers).

Examples: Deciding which server rack in which data center holds which table's data.
🔧 Database Administrators (DBA)
🔑 Users interact only with the External Level. Physical reality is completely hidden by the two layers above it.
Three levels of abstraction introduced in 1970 — External (what users see), Conceptual (logical blueprint), Physical (actual storage).
A

External Schema — Role-Based Views

The external level is all about what different users see when they log in. Not everyone gets the same view — the system shows each user only the data relevant to their role. This is where authorization is enforced.

👥 External Schema — Role-Based Views (University Example)
👨‍🎓 Student View
Can see:
• Own marks & attendance
• Own fee structure
• Course timetable

Cannot see:
• Other students' records
• Faculty salary

READ ONLY
👩‍🏫 Faculty View
Can see & do:
• Enter / update student marks
• Apply for leave
• View own salary
• Course assignments

Cannot see:
• Other faculty salary

READ + WRITE (limited)
👔 Dean / Admin View
Can see & do:
• All student records
• All faculty records
• Department reports
• Budget & financials

 
FULL READ + WRITE
The same database — three completely different views. Authorization at the External Level ensures users see only what they're permitted to see.

Conceptual Schema — The Logical Blueprint

The conceptual schema is like the architectural blueprint of a building — before construction begins, an architect draws out every room, wall, and connection. Similarly, a database designer creates the conceptual schema before writing a single line of code — defining every table, every column, and every relationship.

The conceptual schema defines all tables, columns, data types, and foreign-key relationships — the full logical blueprint of the database.
C

Physical Schema — Where Data Actually Lives

The physical schema is the lowest level — it decides exactly where and how data is stored on hardware. This is the domain of Database Administrators (DBAs), who have full control over servers, disks, and storage strategies.

🔧 Physical Schema — DBA's Domain
🔧
Database Administrator (DBA)
Decides:
• Which hard disks to use
• File names & file sizes
• Index structures
• Fragmentation strategy
Storage models:
• Centralized — all data on one server
• Distributed — data spread across multiple global servers
• Replication, backups, RAID
The DBA operates at the physical level — completely invisible to users and even to most developers.
4

The Biggest Trick: Tables vs. Files

🤔 Most common confusion: Students think data is stored as tables on the hard disk. It is not. Data is stored as files. The DBMS converts those files into tables when you query them.
💡 What You See vs What's Actually on Disk
👁️ What the User Sees (Table)

When you search IRCTC for trains:

Train No.NameDepartsSeats
12301Rajdhani16:5545
12951Mumbai Raj.17:0012

Neat rows & columns — easy to read

💾 What's on the Hard Disk (Files)

Physical reality on the server:

📄 trains_001.ibd    2.4 GB
📄 trains_002.ibd    2.1 GB
📄 bookings.ibd     8.7 GB
📄 users.ibd        1.2 GB

Binary files — not human-readable

⚙️ The DBMS reads the physical files and converts them into the table format you see — guided by the rules in the Conceptual and Physical schemas.
Data is NEVER stored as tables on disk. Tables are a logical abstraction created by the DBMS on top of physical files.

Schema — Quick Recap
Definition:The structure used to store data. In RDBMS = 2D tables (rows + columns).
Also defines:Column characteristics — e.g. roll number is INTEGER of 2-byte or 4-byte size.
Introduced:Three-Schema Architecture introduced in 1970.

Main Motive — Data Independence & Abstraction
Data Abstraction:Hiding physical storage details from the user. Example: Gmail hides server location and file format — you just see your inbox.

Goal:Users can access data without knowing how or where the DB operates physically.
3
Three Levels of Abstraction
Level 3 — External Schema (View Level):
How data is shown to users. Role-based views — each user sees only their permitted data. Enforces authorization (read vs read-write). Built by front-end devs (PHP, Java, HTML).
Level 2 — Conceptual Schema ⭐ (Logical Level):
The complete logical blueprint — all tables, columns, data types, relationships. Like a building's architectural blueprint. Uses E-R Model. Designed by DB designers.
Level 1 — Physical Schema (Internal Level):
Where and how data is physically stored — actual disks, file names, file sizes, fragmentation, centralized vs distributed. Managed by DBA.
A
External Schema — Key Points
What it does:Shows each user a customized view of the database based on their role.
University example:Student → sees own marks/fees (read-only). Faculty → enters marks, applies leave. Dean → full access.
Apps example:Paytm, Amazon, university portals — what you see on the front page is the external schema in action.
Authorization:Student can READ marks but cannot UPDATE them. Different roles = different permissions.
Who builds it:Front-end / interface designers using PHP, Java, HTML.
B
Conceptual Schema — Key Points
What it is:The complete logical blueprint of the entire database.
Defines:What tables exist, their columns, data types (roll no = INT, name = VARCHAR), and relationships between tables (FK links).
Analogy:Like an architectural blueprint — the full design plan given to a builder before construction begins.
Tool used:E-R (Entity-Relationship) Model — primary tool for conceptual schema design.
Who designs it:Database Designers — they focus on logical structure without worrying about physical storage location.
C
Physical Schema — Key Points
What it controls:Actual hard disks, file names, file sizes, fragmentation, storage distribution.
Centralized:All data stored in one physical location / server.
Distributed:Data spread across multiple servers in different global locations.
Who manages it:DBA (Database Administrator) — has complete control and decides specific drives and locations.
Hidden from:Users, developers, and even database designers. Completely invisible above the physical level.
4
The Key Trick — Tables vs. Files
Common myth:Data is stored as tables on the hard disk. ❌ WRONG.
Reality:Data is stored as binary files on disk (e.g. trains.ibd, bookings.ibd). Not as rows and columns.
What DBMS does:Acts as a layer on top of files — reads the binary files and presents them as neat tables using the rules defined in the Physical and Conceptual schemas.
Example:IRCTC search → you see a table of trains. The server actually read binary .ibd files. The DBMS converted them to the table you see.
