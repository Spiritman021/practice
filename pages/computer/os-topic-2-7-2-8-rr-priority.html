2.7

Round Robin (RR) Scheduling

🔵 Round Robin — Algorithm at a Glance
Selection CriteriaReady Queue FIFO — processes served in arrival sequence
ParametersTime Quantum (TQ) — maximum continuous CPU time slice
ModePreemptive — preempted automatically when TQ expires
StarvationNo — fair sharing, every process gets a turn in cycles
Context SwitchHigh — frequent switching causes overhead
💡 Why Ready Queue is Critical in RR: You MUST draw a separate Ready Queue. When a process finishes its time slice, two things happen at that exact timestamp: (1) New processes arrive in the queue, (2) The preempted process goes back to the queue. **Always place new arrivals first, then the preempted process.**

1
At t=0: Only P1 arrived. Ready Queue = [P1]. Extract P1 → runs 0 → 2. Remaining BT of P1 = 3.
Ready Queue t=0:P1
2
At t=2: P1 slice expires. Check new arrivals from t=0 to t=2: P2 (AT=1) and P3 (AT=2) have arrived. Add them first: [P2, P3]. Then append preempted P1 to the end. Ready Queue = [P2, P3, P1].
Extract P2 → runs 2 → 4. Remaining BT of P2 = 2.
Ready Queue t=2:P2P3P1
3
At t=4: P2 slice expires. Check new arrivals: P4 (AT=4) just arrived. Add P4: [P3, P1, P4]. Then append preempted P2 to the end. Ready Queue = [P3, P1, P4, P2].
Extract P3 → runs 4 → 6. Remaining BT of P3 = 0. P3 completes at t=6.
Ready Queue t=4:P3P1P4P2
4
At t=6: P3 completed. No new arrivals. Ready Queue = [P1, P4, P2]. P3 is not appended since BT=0.
Extract P1 → runs 6 → 8. Remaining BT of P1 = 1. Append preempted P1. Ready Queue = [P4, P2, P1].
Ready Queue t=6:P1P4P2
5
At t=8: P1 slice expires. Ready Queue = [P4, P2, P1].
Extract P4 → P4 only needs 1 unit! Run P4 8 → 9. **Do not run P4 for 2 units.** P4 remaining = 0. P4 completes at t=9. Ready Queue = [P2, P1].
Ready Queue t=8:P4P2P1
6
At t=9: Extract P2 → needs 2 units. Run P2 9 → 11. P2 remaining = 0. P2 completes at t=11. Ready Queue = [P1].
Ready Queue t=9:P2P1
7
At t=11: Extract P1 → needs 1 unit. Run P1 11 → 12. P1 remaining = 0. P1 completes at t=12. Queue empty.
Ready Queue t=11:P1

2.8

Preemptive Priority Scheduling

👾 Preemptive Priority — Algorithm at a Glance
Selection CriteriaPriority Value — highest priority gets CPU first
Priority OrderSpecified in question (e.g., Higher number = Higher priority)
ModePreemptive — preempt immediately if a higher priority process arrives
StarvationYes — low priority processes may starve indefinitely
Solution to StarvationAging — gradually increase priority of waiting processes

Rule: Higher priority value means higher priority. Preemptive: compare priorities at every new arrival.
1
At t=0: Only P1 arrived. Run P1 for 1 unit. P1 remaining BT = 4.
2
At t=1: P2 (Pri=20) arrives. Compare P1 (Pri=10) vs P2 (Pri=20). P2 has higher priority → Preempt P1, run P2. Run P2 for 1 unit. P2 remaining BT = 3.
3
At t=2: P3 (Pri=30) arrives. Compare ready: P1(10), P2(20), P3(30). P3 has highest priority → Preempt P2, run P3. Run P3. No arrivals on t=3. P3 runs 2 → 4. P3 completes at t=4.
4
At t=4: P4 (Pri=40) arrives. Ready: P1(10), P2(20), P4(40). P4 has highest priority → Run P4. BT=1 → runs 4 → 5. P4 completes at t=5.
5
At t=5: All processes have arrived. Remaining: P1(Pri=10, rem=4), P2(Pri=20, rem=3). P2 has higher priority → Run P2 to completion. Runs 5 → 8. P2 completes at t=8.
6
At t=8: Only P1 remains. Run P1 to completion. Runs 8 → 12. P1 completes at t=12.

🔗 Priority Tie-Breaking Rules: If priorities are equal: (1) check arrival times (who came first gets CPU), (2) if arrival times also equal: check Process ID (lower ID gets CPU first, e.g., P1 before P2).
