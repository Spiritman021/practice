Integrity Constraints

The four rules that guard database quality — Domain, Entity, Referential, and Key constraints — and how they keep your data accurate and consistent.

GATEUGC NET IBPS SO ITB.Tech / BCA
1

What are Integrity Constraints?

Integrity constraints are a set of rules applied to a database. When you create tables and perform operations like inserting, updating, or deleting data, you cannot just input random or meaningless data. Constraints act as a strict gatekeeper — only valid, well-structured data gets in.

Think of constraints like a form validator on a website. If you try to enter your age as "-5" or leave a required field blank, the system rejects it immediately. Integrity constraints do exactly the same thing at the database level.

🔒 Purpose: Integrity constraints maintain the accuracy, consistency, and reliability of data in the database. They preserve overall data quality and prevent garbage data from corrupting the system.
🗺️ Overview — Four Types of Integrity Constraints
A  —  Domain Constraint CHECK
Based on: Column Data Type
Defines valid values for each column. Prevents invalid entries like negative age or wrong-length phone numbers.
B  —  Entity Integrity PRIMARY KEY
Based on: Primary Key
Every table must have a primary key. That key must be UNIQUE and NOT NULL — no exceptions.
C  —  Referential Integrity FOREIGN KEY
Based on: Foreign Key
Any value in a child table's FK column must already exist in the parent table. No orphan references allowed.
D  —  Key Constraints CANDIDATE KEY
Based on: Candidate Key
At least one attribute (or combination) must hold completely unique values across all records in the table.
Four constraints — each targeting a different aspect of data quality: column values, row identity, table relationships, and uniqueness.
A

Domain Constraints

A domain constraint is tied to the attributes (columns) of a table. When you define a column, you assign it a data type (domain) — like INTEGER for numbers or VARCHAR for text. The domain constraint ensures that only values matching that domain are accepted.

But data types alone aren't enough. You also need business rules — for example, age must be greater than zero, a mobile number must be exactly 10 digits. In SQL, these custom domain rules are implemented using the CHECK constraint.

Domain constraints block invalid column values at the point of entry — before they can corrupt the database.

Here is how domain constraints are implemented in SQL using the CHECK keyword:

CREATE TABLE Student (
  RollNo    INTEGER      PRIMARY KEY,
  Name      VARCHAR(50)  NOT NULL,
  Age       INTEGER      CHECK (Age > 0),           -- domain constraint
  Phone     CHAR(10)     CHECK (LENGTH(Phone) = 10)  -- exactly 10 digits
);
B

Entity Integrity Constraints

Entity integrity is all about the Primary Key. Every table must have a primary key so that each row can be uniquely identified. Without it, you can't tell two identical-looking records apart, and the entire relational model breaks down.

📌 Two mandatory properties of a Primary Key:
1. UNIQUE — No two rows can have the same primary key value.
2. NOT NULL — A primary key column can never be empty (NULL).

C

Referential Integrity Constraints

Referential integrity is built on the Foreign Key relationship between two tables: a Parent (Master) table and a Child table. The rule is simple but strict: any value entered into the child table's FK column must already exist in the parent table.

If you try to insert a reference in the child table that doesn't exist in the parent table — that's a referential integrity violation, and the database will reject the operation.

D

Key Constraints

Key constraints are related to the Candidate Key concept. The rule is: in any table, there must be at least one attribute (or combination of attributes) that contains completely unique values for every single record.

Certain real-world attributes are inherently unique — they naturally serve as candidate keys for any individual. These are the building blocks of key constraints.

🗝️ Key Constraints — Naturally Unique Attributes (Candidate Keys)
🪪

Unique per licensed driver in a region
📱

One number registered to one person
📧
Email ID
Unique per account in any system
Key Constraint Rule: At least one of these unique attributes (or a combination of columns) must exist in every table. This attribute becomes a Candidate Key — a candidate for being chosen as the Primary Key. From all candidate keys, one is selected as the official Primary Key.
Any attribute that is inherently unique across all records qualifies as a Candidate Key. Key Constraints require at least one such attribute per table.

What are Integrity Constraints?
Definition:A set of rules applied to a database to ensure data quality during insert, update, and delete operations.
Purpose:Prevent random or invalid data from entering. Maintain accuracy, consistency, and reliability.
Analogy:Like a form validator on a website — bad input is rejected before it can cause damage.
Types (4):Domain, Entity Integrity, Referential Integrity, Key Constraints.
A
Domain Constraint
Based on:Column attributes and their data types (domain).
What it does:Ensures column values match the defined data type AND custom business rules.
SQL keyword:CHECK constraint.
Example 1:CHECK (Age > 0) — rejects -5, 0, or text in Age column.
Example 2:CHECK (LENGTH(Phone) = 10) — ensures phone is exactly 10 digits.
Data types used:INTEGER, VARCHAR, CHAR, DATE, FLOAT — each is a domain constraint in itself.
B
Entity Integrity Constraint
Based on:Primary Key.
Rule:Every table must have at least one primary key to uniquely identify each row.
Two properties:UNIQUE — no two rows can share the same PK value. NOT NULL — PK column can never be empty.
Violation example:Inserting RollNo = 102 when 102 already exists → rejected. Inserting RollNo = NULL → rejected.
Why it matters:Without a PK, two identical rows can't be told apart — the relational model breaks down.
C
Referential Integrity Constraint
Based on:Foreign Key (FK).
Tables involved:Parent (Master) table and Child table.
Rule:Any FK value in the child table must already exist in the parent table's PK column. No orphan references.
Violation example:Student has DeptID = 'D99'. If 'D99' doesn't exist in Department table → INSERT rejected.
Real-world:Order must reference an existing Customer. Blog post must reference an existing Author.
D
Key Constraints
Based on:Candidate Key.
Rule:Every table must have at least one attribute (or attribute combination) with completely unique values across all records.
Examples:Aadhaar number, Driving License, Phone number, Email ID — all inherently unique.
Candidate Key:Any attribute that is naturally unique qualifies. One is chosen as Primary Key; the rest remain as alternate keys.
Composite key:A combination of two attributes can also form a unique key — e.g. (FirstName + DOB) might be unique together.
