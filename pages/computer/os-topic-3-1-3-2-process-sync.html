3.1

Process Synchronization — Why It Is Needed

▶ Two Modes of Process Execution
🔒 Serial Mode (Sequential)
Processes execute one by one.
P2 cannot start until P1 finishes.
📈 Like an ATM — only one customer at a time.
No synchronization needed.
⚡ Parallel Mode (Concurrent)
Many processes run simultaneously.
Time-sharing / multiprogrammed systems.
📈 Like IRCTC — thousands at once.
Synchronization is critical!
▶ DIAGRAM 3.1-A — Process Classification
All Processes in a Parallel System
👥 Independent Processes
Nothing shared between them.
Execution of one does NOT affect other.

Example: Transacting on SBI server + transacting on HDFC server simultaneously — two completely separate servers.
✅ No synchronization needed
🤝 Cooperative Processes
Share something common.
Execution of one CAN affect other.

Example: IRCTC booking — thousands sharing the same ticket DB simultaneously.
⚠ Synchronization REQUIRED
Only Cooperative processes need synchronization. Independent processes run in total isolation.
▶ DIAGRAM 3.1-B — What Can Cooperative Processes Share?
📋
Variable
A global/shared variable that both processes can read and write
💾
Memory / Buffer
A shared region in RAM that both processes access
📄
Code
Common code sections (like shared library functions) used by both
📷
Resources
Hardware: CPU, Printer, Scanner, Disk — shared hardware devices
If any one of these is shared, the processes are Cooperative and must be synchronized.

Race Condition — Solved Example (P1 & P2)

▶ DIAGRAM 3.1-C — Code for P1 and P2 (Shared Variable = 5)
// Process P1
int shared = 5; // GLOBAL
// Local variable
int X;
// Instructions
I1: X = shared; // X=5
I2: X++; // X=6
I3: sleep(1); // PAUSE/PREEMPT
I4: shared = X; // shared=6?
// Process P2
int shared = 5; // GLOBAL (same)
// Local variable
int Y;
// Instructions
I1: Y = shared; // Y=5
I2: Y--; // Y=4
I3: sleep(1); // PAUSE/PREEMPT
I4: shared = Y; // shared=4?
📝 Key: shared is the common/global variable (initialized to 5). X and Y are local to each process. sleep(1) simulates preemption. Expected answer: 5+1−1 = 5. But race condition gives wrong answers!
Both processes share the same variable "shared". Without synchronization, the final value is unpredictable.
▶ DIAGRAM 3.1-D — Execution Trace: If P1 Starts First (Answer = 4, WRONG)
P1 → I1
X = shared → X = 5 (shared was 5). Local X gets value 5.
P1 → I2
X++ → X = 6. Local X is now 6. shared is still 5 (not updated yet).
P1 → I3
sleep(1) → P1 is PREEMPTED! CPU saves P1 state in PCB (X=6, at I4 next). CPU switches to P2.
P2 → I1
Y = shared → Y = 5. shared is still 5 (P1 never wrote back). Local Y gets 5.
P2 → I2
Y-- → Y = 4. Local Y is now 4.
P2 → I3
sleep(1) → P2 is PREEMPTED! CPU saves P2 state in PCB (Y=4, at I4 next). P1's 1s sleep is done → CPU resumes P1.
P1 → I4
shared = X → shared = 6. P1 loads X=6 from its PCB and writes to shared. P1 terminates.
P2 → I4
shared = Y → shared = 4. P2 loads Y=4 from its PCB and overwrites shared. P2 terminates.
Final value of shared
4
✘ WRONG! (Expected: 5)
If P2 starts first instead
6
✘ Also WRONG! (Expected: 5)
Both execution orders give wrong results. The expected answer (5+1-1=5) is never achieved without synchronization.
⚡ What is Race Condition?
DefinitionWhen multiple processes access & modify a shared resource concurrently, and the final result depends on the ORDER of execution
In this example4 and 6 are "racing" — whichever process writes last wins, but both answers are wrong
Root CauseProcesses are cooperative (shared variable) but NOT synchronized
SolutionsSemaphores, Peterson's Solution, Mutex Locks, Monitors (covered in next topics)

Producer-Consumer Problem

🚪 Producer-Consumer — Setup
Two ProcessesProducer (creates items) and Consumer (uses items) running in parallel (cooperative)
Shared: BufferA fixed-size circular buffer of n=8 slots (indices 0 to 7) shared by both
Shared: countGlobal variable tracking how many items are currently in buffer. Starts at 0.
in pointerProducer's pointer — shows NEXT EMPTY slot where producer will place item
out pointerConsumer's pointer — shows NEXT ITEM to consume from buffer
Circular bufferUses mod n: when in/out reaches n-1, next position wraps back to 0
▶ DIAGRAM 3.2-A — Producer and Consumer Code
// PRODUCER
int count = 0; // global, shared
buffer[n]; // shared, n=8
void producer(void) {
int item_p;
while (true) {
produce_item(item_p);
while(count==n) ;
// buffer full, wait
buffer[in] = item_p;
in = (in+1) % n;
count = count + 1;
}
}
// CONSUMER
int count = 0; // same global
buffer[n]; // same buffer
void consumer(void) {
int item_c;
while (true) {
while(count==0) ;
// buffer empty, wait
item_c = buffer[out];
out = (out+1) % n;
count = count - 1;
}
}
Producer guard: while(count==n) = Buffer full, wait • Consumer guard: while(count==0) = Buffer empty, wait.
The highlighted lines count=count+1 and count=count-1 are the CRITICAL SECTIONS causing race condition.
Pink/yellow highlighted lines are where the race condition happens. Both read and write the shared "count" variable.

Circular buffer rule: in = (in+1) mod 8 → after slot 7, wraps to 0. Same for out. This prevents array overflow.
Buffer with 3 items (X1, X2, X3). in points to next empty slot (3). out points to next item to consume (0). count=3.
▶ DIAGRAM 3.2-C — Micro-Instructions: How CPU Executes count = count + 1
// High-level C instruction:
count = count + 1;
// CPU converts to 3 micro-instructions:
i1: LOAD Rp, M[count]
Load count value from memory into register Rp
i2: INCR Rp
Increment register Rp by 1 (in CPU, not in memory yet)
i3: STORE Rp, M[count]
Write Rp back to memory at count address
// Similarly, count = count - 1 becomes:
i1: LOAD Rc, M[count]
i2: DECR Rc
i3: STORE Rc, M[count]
⚠ Critical Insight: The update only reaches memory at i3 (STORE). If a process is preempted between i1-i2 or i2-i3, the other process sees the OLD value of count in memory.
Every simple C statement breaks into multiple CPU micro-instructions. Preemption between i1 and i3 is the source of the race condition.
▶ DIAGRAM 3.2-D — Race Condition Case (Initial State: count=3, 3 items in buffer)
Starting state: Buffer has 3 items (X1, X2, X3). count=3. in=3, out=0. Producer wants to add X4. Consumer wants to consume X1.
Producer → i1
LOAD Rp, M[count] → Rp = 3 (count is 3 in memory). Rp is producer's register.
Producer → i2
INCR Rp → Rp = 4. Only the register is updated. M[count] still shows 3!
⚡ PREEMPT
Producer is preempted BEFORE i3! PCB saves Rp=4. M[count] is still 3. CPU switches to Consumer.
Consumer → i1
LOAD Rc, M[count] → Rc = 3. Reads count from memory (still 3, Producer never wrote back!).
Consumer → i2
DECR Rc → Rc = 2. Rc is decremented. M[count] still 3.
⚡ PREEMPT
Consumer is preempted BEFORE i3! PCB saves Rc=2. CPU resumes Producer (its sleep done).
Producer → i3
STORE Rp, M[count] → M[count] = 4. Writes Rp=4 to memory. count is now 4. Producer terminates.
Consumer → i3
STORE Rc, M[count] → M[count] = 2. Writes Rc=2 to memory. count is now 2. Consumer terminates.
Final count value
2
✘ WRONG! Buffer has 3 items but count says 2
Expected count value
3
3 items + 1 produced − 1 consumed = 3
Consumer's i3 (STORE Rc=2) executed AFTER Producer's i3 (STORE Rp=4) and OVERWROTE it. Final wrong count = 2.
!
Execution Order: Producer(i1) → Producer(i2) → [Preempt] → Consumer(i1) → Consumer(i2) → [Preempt] → Producer(i3) → Consumer(i3). This specific interleaving causes the race condition.
!
Why count=2 is wrong: We had 3 items, added X4 (+1=4), removed X1 (-1=3). Expected count=3. But Consumer's STORE ran last and overwrote with Rc=2.
▶
Key Insight: Consumer's Rc was loaded when count was 3 (before Producer's i3 updated it to 4). So Consumer decremented 3 to 2 and stored 2, wiping out Producer's update.
✓
Solution: The critical section (count = count ± 1) must be made atomic — only one process can access it at a time using Semaphores, Mutex Locks, or Monitors.
⭐ GATE Point: The Producer-Consumer problem is a classic example of a Critical Section Problem. The "critical section" is the code where shared data is accessed and modified. Any examination of count = count+1 or count = count-1 using micro-instructions (LOAD-INCR/DECR-STORE) is a standard GATE/UGC NET question pattern.

Process Synchronization — Key Points
Serial ModeOne process at a time, no interference. Like ATM queue. No sync needed.
Parallel ModeMultiple processes simultaneously. Time-sharing systems. Sync required for cooperative processes.
Independent ProcessNothing shared. Execution of one does NOT affect other. No sync needed.
Cooperative ProcessShare variable / memory / code / resources. Execution of one CAN affect other. Must synchronize.
Race ConditionFinal result depends on execution ORDER. Multiple answers possible, all wrong.
P1/P2 Example (shared=5)If P1 first: shared = 4 (wrong). If P2 first: shared = 6 (wrong). Expected: 5.
★ Root Causesleep(1) causes preemption. P2 reads old value of "shared" before P1 writes back.
★ SolutionsSemaphore, Peterson's Solution, Mutex Locks, Monitors.

Producer-Consumer Problem
SetupProducer + Consumer sharing buffer (n=8) and global count. Both parallel (cooperative).
in pointerNext empty slot for producer. Updated: in = (in+1) % n.
out pointerNext item for consumer. Updated: out = (out+1) % n.
countItems in buffer. Shared by both. count++ by producer, count-- by consumer.
Buffer Full guardProducer: while(count == n); — infinite loop if buffer full.
Buffer Empty guardConsumer: while(count == 0); — infinite loop if buffer empty.
Micro-instructionscount=count+1 → LOAD Rp, INCR Rp, STORE Rp. 3 separate machine operations!
★ Race Condition CaseProd(i1,i2) → preempt → Cons(i1,i2) → preempt → Prod(i3) → Cons(i3). Count=2 but buffer has 3.
★ Why wrong?Consumer loaded count=3 BEFORE producer wrote count=4. Consumer's STORE(Rc=2) overwrites producer's update.
★ Critical SectionThe count modification code is the Critical Section. It must be atomic (uninterruptible).
