3.3

Printer-Spooler Problem

📷 Printer-Spooler — Background
SetupNetwork with 1 shared printer and many users (processes) wanting to print simultaneously
Why Spooler?Printer is a slow peripheral device. A spooler (a program) queues print jobs and feeds them to the printer one by one
Spooler DirectoryA directory (like a queue) in memory where print jobs wait. Printer picks them sequentially.
IN variableA shared global variable that always points to the next empty slot in the spooler directory
CooperativeAll processes share the same spooler code and the IN variable → they are cooperative

▶ DIAGRAM 3.3-B — The 4-Line Spooler Code (Every Process Must Execute This)
// Spooler insertion code (run by EVERY process)
I1: LOAD Ri, M[IN]  ← Load IN value into register Ri
I2: STORE FileName, SD[Ri]  ← Put file in spooler at position Ri
I3: INCR Ri  ← Increment register (next slot)
I4: STORE Ri, M[IN]  ← Write updated IN back to memory
Ri = dedicated register for process i (R1 for P1, R2 for P2). Each process has its own register but they ALL share the same IN variable in memory.
All processes execute the same 4 instructions. The critical shared resource is the IN variable and the spooler directory slots.
C1

Case 1 — Normal Case (Single Process, Works Fine)

I1
LOAD R1, M[IN]: IN=0 (spooler empty). Ri = R1 = 0. Load 0 into P1's dedicated register R1.
I2
STORE f1.DOC, SD[R1]: R1=0. Place file f1.DOC in spooler directory slot 0. SD[0] = f1.DOC.
I3
INCR R1: R1 goes from 0 → 1. Now R1 points to the next available slot.
I4
STORE R1, M[IN]: Write R1=1 back to IN. Now IN=1. Printer can pick f1.DOC. ✅ No problem.

▶ DIAGRAM 3.3-D — Race Condition Execution Trace (Data Loss!)
P1 → I1
LOAD R1, M[IN] → R1 = 3. P1 reads IN=3 into its dedicated register R1.
P1 → I2
STORE f4.DOC, SD[R1=3] → SD[3] = f4.DOC. P1 places f4.DOC at slot 3.
P1 → I3
INCR R1 → R1 = 4. Register incremented. IN in memory is still 3 (not updated yet!).
⚡ PREEMPT P1
P1 preempted after I3! PCB saves R1=4. M[IN] is still 3. CPU switches to P2.
P2 → I1
LOAD R2, M[IN] → R2 = 3. P2 reads IN from memory — still 3! P1 never wrote back.
P2 → I2
STORE f5.DOC, SD[R2=3] → SD[3] = f5.DOC. P2 writes f5.DOC to slot 3 — OVERWRITING f4.DOC!
P2 → I3
INCR R2 → R2 = 4.
P2 → I4
STORE R2=4, M[IN] → IN = 4. P2 terminates successfully.
P1 resumes → I4
STORE R1=4, M[IN] → IN = 4 (same value, no harm to IN). P1 terminates. But f4.DOC is gone!
⚠ Final Spooler Directory State
SD[0]: f1.DOC
SD[1]: f2.DOC
SD[2]: f3.DOC
SD[3]: f5.DOC 🔴
SD[4]: empty ← IN
f4.DOC is LOST! Both P1 and P2 got R=3 from memory and wrote to SD[3]. P2 wrote last, so f5.DOC stays. P1's f4.DOC was silently overwritten. P1's user will never know their document was not printed. This is Data Loss — the worst kind of race condition.
Execution order: P1(I1,I2,I3) → preempt → P2(I1,I2,I3,I4) → P1(I4). Both processes read IN=3 and write to SD[3]. Only f5.DOC survives.
⭐ Exam Point: The Printer-Spooler problem is a classic example of a Race Condition causing Data Loss. The fix is to make the 4-instruction code sequence a Critical Section — only one process can execute it at a time. This is achieved using Semaphores, Locks, or Monitors.
3.4

Critical Section — Definition & 4 Conditions

🔐 Critical Section — Core Concepts
DefinitionPart of a program where shared resources (variable, memory, code, hardware) are accessed by concurrent cooperative processes
Critical SectionThe code segment that must be executed by only ONE process at a time
Non-Critical SectionIndependent code not shared with any other process — can run freely without restriction
Entry SectionCode that must run BEFORE entering Critical Section (like acquiring a lock)
Exit SectionCode that runs AFTER leaving Critical Section (like releasing a lock)

Analogy: Entry Section = locking a washroom door before entering. Critical Section = using the washroom. Exit Section = unlocking the door when done. No one else can enter while it's locked — that's Mutual Exclusion!
Entry/Exit sections are the synchronization code (lock/unlock). Only one process can be in the Critical Section at any time.
4C

4 Conditions Every Synchronization Solution Must Satisfy

▶ DIAGRAM 3.4-B — Overview: Primary vs Secondary Conditions
🔴 PRIMARY (Mandatory)
1. Mutual Exclusion
Only one process in CS at a time
2. Progress
Empty CS must not block interested processes
If either fails, the solution is INVALID
🟢 SECONDARY (Important)
3. Bounded Wait
No process waits indefinitely
4. No Hardware Assumption
Solution works on any system/speed
Try to follow; failure not fatal but problematic
Primary conditions are MANDATORY. Secondary are highly important. All 4 together make a perfect synchronization solution.
🔐 Condition 1: Mutual Exclusion (PRIMARY)
RuleIf Process P1 is inside Critical Section, then NO other process (P2, P3, ...) can enter Critical Section
AnalogyWashroom with a lock. If P1 is inside and locked it, P2 cannot enter even if it wants to.
ImplementationEntry Section must ensure: if CS is occupied, block all other processes trying to enter
ViolationIf P1 is in CS and P2 also enters CS simultaneously → Mutual Exclusion is violated → solution is INVALID
NoteNo constraint on which process enters FIRST — either P1 or P2 can win. But once one is in, the other MUST wait outside.
▶ Condition 2: Progress (PRIMARY)
RuleIf CS is EMPTY and one or more processes WANT to enter, the decision on who enters must happen in finite time — no outside process can block the entry
Violation ScenarioCS is empty. P1 wants to enter. P2 does NOT want to enter. But P2's entry code BLOCKS P1 from entering. This is a Progress violation!
Real-world analogy"Himself doesn't want to work, but doesn't let others work either." — exactly what violates Progress.
Difference from MEMutual Exclusion: CS is OCCUPIED, block others. Progress: CS is EMPTY, interested processes must be ALLOWED in, not blocked.
Violation consequenceNeither process can use CS even though it's free → deadlock-like state → CS never executes.
⏳ Condition 3: Bounded Wait (SECONDARY)
RuleThere must be an UPPER BOUND on the number of times other processes can enter CS after a process has requested entry and before that request is granted
What is allowedP1 uses CS 10 times, P2 uses CS only once — this is OK (different usage counts are fine)
What is NOT allowedP1 uses CS infinitely, P2 gets to use only once or twice → P2 starves → Bounded Wait violated
AnalogyP1 repeatedly re-entering a shop while P2 is stuck waiting outside forever. There should be a maximum queue skip limit.
Relation to StarvationViolation of Bounded Wait = Starvation. One process waits indefinitely while others keep using CS repeatedly.
⚙ Condition 4: No Assumption on Hardware / Speed (SECONDARY)
RuleSynchronization solution must NOT make any assumptions about hardware (32-bit vs 64-bit), processor speed (1GHz vs 4GHz), or operating system (Linux vs Windows)
Must be portableSolution should run on any hardware, any OS, any processor speed without modification
Violation example"This solution only works on Linux at 3GHz" → Not a valid universal synchronization mechanism
GoalUniversal, portable solution that works across all environments equally well
▶ TABLE 3.4-C — Quick Comparison of All 4 Conditions

Conditions 1 & 2 are mandatory. If any solution fails Mutual Exclusion or Progress, it is outright rejected. Conditions 3 & 4 are important for fairness and portability.
⭐ GATE / Exam Point: These 4 conditions are asked very frequently. Remember: Mutual Exclusion + Progress = Primary (Mandatory). Bounded Wait + No HW Assumption = Secondary. For any given synchronization algorithm (Semaphore, Peterson's, TSL, etc.) you may be asked: "Which of the 4 conditions does it satisfy?" Printer-Spooler Problem
What is Spooler?A program/directory that queues print jobs and feeds them to the (slow) printer one at a time.
IN variableShared global variable pointing to next EMPTY slot in spooler directory.
4 InstructionsI1: LOAD Ri,M[IN] • I2: STORE filename,SD[Ri] • I3: INCR Ri • I4: STORE Ri,M[IN]
Race ConditionBoth P1 and P2 read IN=3. Both write to SD[3]. P2 overwrites P1's file. f4.DOC is lost.
★ Problem typeDATA LOSS — worst type of race condition. File disappears silently with no error message.
★ Execution orderP1(I1,I2,I3) → preempt → P2(I1,I2,I3,I4) → P1(I4). Both get Ri=3, both write to SD[3].
Why IN=4 is fine?Both P1 and P2 compute Ri=4 and write IN=4 (same value). No corruption in IN, but data is lost at SD[3].
FixMake I1-I4 a Critical Section. Use Semaphore/Lock so only one process executes at a time.
3.4
Critical Section & 4 Conditions
Critical SectionPart of program where shared resources/variables are accessed. Must be atomic.
Non-Critical SectionIndependent code. No sharing. Can run freely without synchronization.
Entry SectionCode before CS (lock acquisition). If another process is in CS, this blocks entry.
Exit SectionCode after CS (lock release). Signals other waiting processes that CS is free.
🔴 1. Mutual ExclusionPRIMARY. Only 1 process in CS at a time. Like a washroom lock.
🔴 2. ProgressPRIMARY. Empty CS must not block interested processes. "He won't do it himself, won't let others either" = violation.
🟢 3. Bounded WaitSECONDARY. No starvation. P1 can use CS more than P2, but not infinitely more.
🟢 4. No HW AssumptionSECONDARY. Solution must be portable — work on any OS, any hardware, any CPU speed.
★ Primary vs SecondaryMutual Exclusion + Progress = MANDATORY (solution invalid without them). Bounded Wait + No HW = Important but secondary.
