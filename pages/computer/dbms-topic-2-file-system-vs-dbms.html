The Era Before DBMS — File Systems

Before DBMS became the standard, organizations managed data using File Systems — still present today in operating systems as CIFS (Windows/network) and NFS (Linux/Unix). Data was stored locally in a hierarchical structure of drives, folders, and sub-folders.

File systems work perfectly fine for a single user on a personal laptop. But they completely fall apart when applied to Client-Server architectures, where data sits on centralized servers accessed by thousands of global users simultaneously.

⚠️ Key Problem: A file system has no concept of "serving multiple users simultaneously." The moment you need concurrent access, role-based security, or efficient search — file systems are inadequate.
2

Five Major Disadvantages of File Systems

A

Inefficient Searching & Memory Utilization

File System loads the entire file; DBMS returns only what you asked for

Imagine you want just one train's departure time from the IRCTC database. With a file system, the server might load the entire 25 GB file of all train records just to give you 1 KB of data. This is extremely wasteful in both time and memory.

DBMS solves this with SQL queries — you specify exactly what you want, and the server fetches only that record. The difference is like using the index of a book vs reading the entire book to find one sentence.

B

Lack of Location Independence

FS requires knowing the exact file path; DBMS is fully transparent

In a file system, you must know the exact physical path to your data: C:/server/irctc/2024/trains.dat. This is impossible at scale — users can't be expected to navigate server directories.

With DBMS, location independence is built-in. A user in Delhi opens the IRCTC app, the request travels to a server in Mumbai, data is fetched and returned — entirely transparent to the user. They never see a file path.

❌ File System
User must know: C:/folder1/folder2/data.dat
Physical location must be provided explicitly. Different servers = different paths. No abstraction layer.
✓ DBMS
User opens app → sends request → gets data.
Server could be in Mumbai, Singapore, or the cloud. Fully transparent — no path knowledge needed.
C

Poor Concurrency Control

Simultaneous users cause data chaos in FS; DBMS handles it flawlessly
📌 Concurrency = multiple users reading/writing the same data at the same time. IRCTC handles 2–3 lakh (200,000–300,000) transactions daily, with thousands booking concurrently at peak times.

File systems have no protocols for managing simultaneous access. If two users try to book the last seat at the same moment, both might succeed — creating a double-booking. This is data inconsistency.

DBMS comes with built-in transaction control and locking protocols (shared locks, exclusive locks, MVCC). Only one write transaction can modify a record at a time. All others wait in a controlled queue.

D

Lack of Role-Based Security

OS login = full access in FS; DBMS enforces per-role visibility

Operating systems offer coarse security — one password gets you into the machine, and then you can see everything. This is dangerous for multi-user systems like universities or hospitals.

DBMS provides Hierarchical Role-Based Access Control (RBAC). Roles define what data each user can see or modify. A student can only access their own record. A dean sees department reports. Only the DBA sees everything.

✕ File System: One password = Access to all files on the machine
✓ DBMS: Role defines exactly what data you can see or edit
RBAC makes unauthorized data completely invisible to lower-privileged roles — not just hidden, but inaccessible.
E

High Data Redundancy (Duplicacy)

No prevention in FS; DBMS enforces Primary Key + Foreign Key uniqueness

In a file system, nothing stops you from saving "Ram, Roll No A101" four times in the same file, or copying the same data across multiple files with different names. This leads to storage waste and inconsistent records.

DBMS enforces data integrity constraints: the Primary Key ensures every record is unique. The Foreign Key maintains referential integrity across related tables. Any attempt to insert a duplicate throws an error instantly.

Conclusion — The Backend Standard

In today's world, every major application — Amazon, Google, Flipkart, IRCTC — is built on a Client-Server architecture. The frontend is what users see, but the backend always runs on a DBMS to efficiently handle requests, enforce security, and maintain clean, consistent data.

⚡
Efficiency
SQL fetches only what's needed. No wasteful full-file loading.
🔐
Security
RBAC ensures every user sees only their permitted data.
📏
Integrity
Primary Keys and constraints keep data unique and clean.

Conclusion — One-Line Summary
File systems = built for local, single-user, personal use.
DBMS = built for global, multi-user, Client-Server architectures.

All modern applications (Amazon, Google, Flipkart, IRCTC) use DBMS on the backend. The 5 limitations of file systems (search, location, concurrency, security, redundancy) are all solved by DBMS through SQL, locking protocols, RBAC, and integrity constraints.
