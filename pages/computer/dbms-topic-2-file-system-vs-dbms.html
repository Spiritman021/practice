Detailed Notes: File System vs. DBMS

1. The Era Before DBMS: The File System

  • Before the adoption of Database Management Systems (DBMS), data was primarily managed using File Systems, which are still built into modern operating systems (like CIFS and NFS).
  • In a file system, users store data locally in a hierarchical format using drives, folders, and sub-folders.
  • While file systems are perfectly fine for an individual user managing local files on a personal laptop, they fail when applied to modern Client-Server architectures. In these architectures, data is stored on centralized servers and accessed by thousands of users globally.

2. Five Major Disadvantages of File Systems (and Advantages of DBMS)

A. Inefficient Searching and Memory Utilization

  • File System: If you need to search for a tiny piece of data (e.g., 1 KB of specific train details on the IRCTC network), the file system might fetch the entire 25 GB file containing all train records. This wastes immense time and memory.
  • DBMS: You can write a simple SQL query, and the server will fetch and return only the specific 1 KB of data requested. This makes searching fast and memory utilization highly efficient.

B. Lack of Location Independence

  • File System: To access data, the user must know the exact meta-data and physical location of the file (e.g., C:/folder 1/folder 2).
  • DBMS: Data is completely location-independent. The user does not need to know where the server is physically located. They simply open a web or mobile app, send a request, and the DBMS responds seamlessly without requiring the user to navigate file paths.

C. Poor Concurrency Control

  • Concurrency means multiple users accessing the same data simultaneously. For instance, IRCTC handles 2 to 3 lakh transactions daily, with thousands of users booking concurrently.
  • File System: It lacks the protocols needed to manage simultaneous access safely. If thousands of users try to read or write data at the same time, it leads to severe data inconsistency.
  • DBMS: It is built with proper transaction control and concurrency protocols to manage simultaneous read-write and write-write conflicts flawlessly.

D. Lack of Role-Based Security

  • File System: Operating systems offer very basic security. Once a user logs into a laptop with a password, they generally have full access to all the drives and files.
  • DBMS: It provides Hierarchical or Role-Based Access Control (RBAC). Different users are assigned different roles (e.g., an Admin vs. a regular User, or a Dean vs. a Student in a university). Users can only see the specific data permitted by their role, making unauthorized data invisible to them.

E. High Data Redundancy (Duplicacy)

  • File System: It does not actively prevent data duplicacy. For example, you could save a student named "Ram" four times in the same document, or save identical data in multiple files just by changing the file names.
  • DBMS: It enforces data integrity through constraints like Primary Keys and Foreign Keys. These strict rules ensure that you can only store unique, non-redundant data.

3. Conclusion: The Backend Standard

In today's world, massive applications like Amazon, Google, and Flipkart all rely on a Client-Server architecture. The users sit at the frontend, but the backend always runs on a DBMS to efficiently handle requests, enforce security, and maintain clean data.
