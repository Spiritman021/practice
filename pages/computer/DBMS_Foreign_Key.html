What is a Foreign Key?

A Foreign Key (FK) is an attribute — or a set of attributes — in a table that references the Primary Key of either the same table or another table. It is the mechanism that creates a logical link between two tables in a relational database.

🔗 Main Purpose: The primary role of a Foreign Key is to maintain Referential Integrity — ensuring that relationships between tables remain consistent and that no "orphan" data exists.
🔑 Simple Foreign Key
References a Simple Primary Key — a PK made of just one attribute.

Example: FK references a single Roll_Number column.
🔑🔑 Composite Foreign Key
References a Composite Primary Key — a PK made of two or more attributes combined.

Example: FK references both Student_ID + Course_ID together.

The Rule of Referencing — Data Entry Restriction

A Foreign Key column is not free to accept any random value. It is bound by a strict rule: the value entered in the FK column must already exist in the PK column of the referenced table. If the value doesn't exist there, the insert is rejected.

Important Terminology

📖 Referenced Table vs Referencing Table
👑 Referenced Table  /  Base Table
The table that contains the Primary Key. It is the source of the reference — other tables point to it.

In our example: The Student table is the Referenced Table because it holds the PK (Roll_No) that the Course table points to.

Think of it as the parent or master table.
🔗 Referencing Table
The table that contains the Foreign Key. It is the table doing the referencing — it points to another table's PK.

In our example: The Course table is the Referencing Table because it holds the FK (Roll_No) that points to Student.

Think of it as the child or dependent table.
Referenced = has the PK (parent). Referencing = has the FK (child). The FK always points from child → parent.

How to Implement Foreign Keys in SQL

⚠️ Data Type Rule: The data type of the Foreign Key column must be exactly the same as the data type of the Primary Key it is referencing. If PK is INTEGER, the FK must also be INTEGER. Mismatch = error.
Method 1 DURING CREATE
Define the FK while creating the table using the REFERENCES keyword inline.
Method 2 AFTER CREATE
Add FK to an existing table using ALTER TABLE with a named constraint.

Method 1 — During Table Creation:

-- Step 1: Create the Referenced (parent) table first
CREATE TABLE Student (
  Roll_Number  INTEGER    PRIMARY KEY,
  Student_Name VARCHAR(50),
  Address      VARCHAR(100)
);

-- Step 2: Create Referencing (child) table with FK defined inline
CREATE TABLE Course (
  Course_ID    VARCHAR(10)  PRIMARY KEY,
  Course_Name  VARCHAR(80),
  Roll_Number  INTEGER    REFERENCES Student(Roll_Number) -- FK defined here
);

Method 2 — After Table Creation using ALTER:

-- Table already exists, add FK constraint later
ALTER TABLE Course
ADD CONSTRAINT FK_StudentCourse                     -- name the constraint
FOREIGN KEY (Roll_Number)                          -- FK column in Course
REFERENCES Student(Roll_Number);                  -- PK in Student

Golden Rules & Important Facts

⭐ Golden Rules — Foreign Key Facts for Exams
1
FK references the PK of another (or same) table
The FK value must exist in the PK column of the referenced table. No orphan references allowed.
REFERENTIAL INTEGRITY
2
Data types must match exactly
If the PK is INTEGER, the FK must also be INTEGER. Mismatched data types cause errors at creation time.
SAME DATA TYPE
3
Column names don't have to match
The FK column can have a different name from the PK it references. It's the data type and the REFERENCES keyword that creates the link — not the name.
NAMES DIFFER OK
4
A table can have MULTIPLE Foreign Keys
Unlike Primary Keys (only 1 per table), a table can have many FK columns — each referencing a different table or the same table.
MANY FKs ALLOWED
5
Referenced table must be created first
When creating tables, always create the Referenced (parent) table before the Referencing (child) table. The parent's PK must exist before the child's FK can point to it.
PARENT FIRST
These five rules cover every FK question that appears in GATE, UGC NET, IBPS SO, and placements.

Linking Tables — Student ↔ Course Example
Student table:Roll_No (PK), Student_Name, Address. Roll_No is PK → UNIQUE + NOT NULL.
Course table:Course_ID (PK), Course_Name, Roll_No (FK). Roll_No here is the Foreign Key.
The link:Roll_No in Course table takes reference from Roll_No PK in Student table.
Why needed:Whenever two tables are related, at least one common attribute must link them — that's the FK.

The Referencing Rule
Core rule:FK value must already exist in the PK column of the referenced table before it can be inserted.
Example:Can't assign Roll_No = 10 to a course if student 10 doesn't exist in Student table → INSERT rejected.
Order matters:Insert the parent record (Student) before inserting the child record (Course) that references it.

SQL Implementation — Two Methods
Data type rule:FK data type must exactly match the PK data type it references.

Method 2 (ALTER):ALTER TABLE Course ADD CONSTRAINT FK FOREIGN KEY (Roll_Number) REFERENCES Student(Roll_Number)
Create order:Always create the Referenced (parent) table first, then the Referencing (child) table.
