Normalization
Basics in DBMS

Normalization is the database design technique used to remove or reduce redundancy from tables and avoid insertion, deletion, and update anomalies.

1

What is Normalization?

Normalization is a DBMS technique used to remove or reduce redundancy, also called duplicacy, from a table.

If the same information is stored again and again, the table becomes difficult to insert into, delete from, and update correctly. Normalization divides such data into better-structured tables.
2

Row-Level Duplicacy

Row-level duplicacy means two complete rows are exactly the same. In the example below, row 1 and row 3 repeat the same student data.

Student Table - Duplicate Row Problem
SIDSNameAgeObservation
1A20Original row
2B21Different row
1A20Duplicate of row 1
Primary Key on SID prevents this because a Primary Key is unique and not null.
Fix: Make SID the Primary Key. Since SID cannot repeat, the same student row cannot be inserted again.

4

Anomalies Caused by Redundancy

An anomaly is a problem that appears during a specific database operation. In this table, redundancy creates three common anomalies.

Insertion Anomaly
If the university starts a new course C10 - MBBS, it cannot be inserted unless a student is also inserted, because SID is the Primary Key and cannot be empty.
Deletion Anomaly
If the only student of a course is deleted, the course and faculty information stored in that same row may also be lost accidentally.
Update Anomaly
If salary of faculty F1 changes from 30000 to 40000, it must be updated in every row where F1 appears.
5

Normalization by Decomposition

A simple way to reduce this redundancy is to divide the large combined table into multiple related tables.

Decomposition Diagram

FID, FName, Salary
→
Student
SID, SName
Course
CID, CName
Faculty
FID, FName, Salary
This is one possible decomposition: each main object gets its own table and its own Primary Key.
After Decomposition
  • New course can be inserted directly into Course.
  • New faculty can be inserted directly into Faculty.
  • Student deletion does not delete course/faculty facts.
Update Becomes Cleaner
Salary of F1 is stored once in Faculty. Changing 30000 to 40000 requires one update instead of repeated row updates.
Normalization mainly handles column-level redundancy and protects the database from insertion, deletion, and update anomalies.
