Question Statement

UGC NET July 2018 — Foreign Key and Referential Integrity

Let R1(a, b, c) and R2(x, y, z) be two relations. Attribute a in R1 is a Foreign Key that refers to the Primary Key of R2. Which pair of operations can cause a referential integrity violation?

1. Insert into R1 and Insert into R2
2. Insert into R2 and Delete from R1
3. Delete from R1 and Delete from R2
4. Delete from R2 and Insert into R1
2

Relationship Setup

The key to this question is identifying the referenced table and the referencing table. Since a is a Foreign Key in R1, R1 depends on R2. The Primary Key in R2 is not named in the question, so we can assume x is the Primary Key.

Foreign Key rule: every value in R1.a must already exist in the Primary Key column of R2.
Exam language: R2 is the base / parent / referenced table. R1 is the dependent / child / referencing table.
3

Analyze Each Operation

Insert into R1 a
⚠️ Can Violate
If a new row is inserted in R1, its FK value in a must already exist in R2.x. Inserting a = 7 when 7 is absent from R2 creates an invalid reference.
Insert into R2 b
✓ Safe
R2 is the base table. Adding a new Primary Key value only creates a new possible reference for future rows in R1. It does not break existing FK values.
Delete from R1 c
✓ Safe
Deleting from the referencing table removes a child row. It does not remove any Primary Key from the base table, so no orphan reference is created.
Delete from R2 d
⚠️ Can Violate
If the deleted Primary Key value is still used in R1.a, those rows in R1 will point to a value that no longer exists. That breaks referential integrity.
4

Real-Life Analogy

Imagine R2 as the university's main student registry and R1 as the course enrollment table. A course enrollment can exist only for a student already present in the university registry.

Safe Order of Operations
1
To enroll a student in a course, the student must first exist in the main registry (R2).
2
Trying to insert enrollment directly in course records (R1) for a missing student causes a violation.
3
Deleting a course enrollment from R1 is safe because the main student record still exists.
4
Deleting a student from R2 while course records still point to that student creates orphan data.
Final Answer: operations d and a can cause referential integrity violation. Therefore, Option 4 is correct.

Quick Definitions
TermMeaning
Foreign KeyAttribute in one table that refers to a Primary Key in the same or another table.
Referenced TableParent/base table containing the Primary Key being referred to.
Referencing TableChild/dependent table containing the Foreign Key.
Referential IntegrityEvery FK value must match an existing PK value in the referenced table.
Orphan RecordA child row whose FK value points to a missing parent row.
