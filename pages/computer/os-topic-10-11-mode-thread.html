Topic 10 — User Mode vs Kernel Mode

10A

The Mode Bit — How Hardware Knows Which Mode

Modern CPUs have a special register called the mode bit (also called privilege bit). It has only two values and tells the hardware which level of privilege the currently running code has.

🔲 The Mode Bit — Only Two Values
1
User Mode
Mode Bit = 1
Normal applications run here.
Text editor, MS Office, music player, games, browsers — all run in user mode by default.
Cannot directly access hardware.
⟷
0
Kernel Mode
Mode Bit = 0
OS kernel, device drivers run here.
Has full access to all hardware: CPU registers, RAM, hard disk, monitor, printer, network cards.
Executes privileged instructions.
The processor continuously switches between mode 1 (user) and mode 0 (kernel) during execution. This switching is the heart of the Dual Mode operation.
Mnemonic: Mode bit 1 = User (U-1). Mode bit 0 = Kernel (K-0). Whenever you see "mode bit = 0" → think kernel mode, privileged, OS in control.
10B

Dual Mode Operation — How Switching Works

Whenever a user process needs OS services (file access, device I/O, printing), it invokes a system call. This generates a TRAP (software interrupt) that switches the CPU from user mode to kernel mode. After the kernel finishes, it returns to user mode.

10C

Detailed Example — C Program Reading a File

📋 C Program: Read File → Write File → Print — All Mode Switches
USER MODE (bit=1)
C program starts executing in user mode. int x = 2 + 2; — CPU does this arithmetic. No kernel needed. Mode bit stays 1.
TRAP (1 → 0)
Program calls read(fd, buffer, size); to read a file from hard disk. Hard disk is hardware → needs kernel. TRAP fires → mode bit changes 1 → 0.
KERNEL MODE (bit=0)
OS kernel reads data from hard disk into buffer. Kernel has direct hardware access. Data is fetched. Now kernel calls write() to update the file → still in kernel mode.
RETURN (0 → 1)
System call completes. Kernel returns control to user process. Mode bit changes 0 → 1. Program continues in user mode.
TRAP AGAIN (1 → 0)
Program calls printf("4"); to print to monitor. Monitor is hardware → needs kernel. Another TRAP. Mode 1 → 0.
KERNEL MODE (bit=0)
OS kernel sends "4" to monitor hardware. Output appears on screen. Returns to user mode. Mode 0 → 1.
Key insight: 2+2=4 (arithmetic) does NOT need kernel. But printf("4") (printing to monitor) DOES need kernel because the monitor is hardware under OS control.
10D

Bank Analogy — Why Dual Mode is Necessary

🏦 Bank Analogy — User Mode vs Kernel Mode
👤 You (Customer) = User Mode
  • • You enter the bank (start application)
  • • You fill a withdrawal form (make a system call)
  • • You hand the form through the small window
  • • You CANNOT go behind the counter yourself
  • • You wait for the clerk to process it
  • • Clerk gives you money (kernel returns result)
  • • You leave with money (return to user mode)
⚙️ Bank Clerk/PO = Kernel Mode
  • • Works behind the counter (kernel mode area)
  • • Has access to the vault (hardware/resources)
  • • Processes the withdrawal form (executes sys call)
  • • Verifies identity and balance (checks permissions)
  • • Accesses the drawer (disk/hardware access)
  • • Gives money through the window (returns data)
  • • Controls all resources in the bank (OS manages all)
Why can't you just go behind the counter? If users could directly access hardware (like going behind the counter and taking money from the vault), there would be NO security. Any process could overwrite any other process's data, crash the system, steal information, or take full control. The dual mode system prevents this — user mode code literally CANNOT execute privileged instructions. Hardware enforces this.
The "small window" between you and the clerk = the system call interface. The form you fill = the system call parameters. The entire bank layout = the kernel-managed hardware resources.
⭐ Key exam points:
• Mode bit = 1 → User Mode (less privileged)
• Mode bit = 0 → Kernel Mode (fully privileged)
• Switching from user → kernel mode: TRAP (software interrupt) is generated
• Switching from kernel → user mode: after system call returns
• Dual mode = the two-mode design (user + kernel) — essential for OS security and resource management
• 2+2=4 (arithmetic) = stays in user mode. printf() = needs kernel mode (monitor is hardware)

Topic 11 — Process vs Thread

11A

Definitions — Heavy vs Lightweight

⚙️ Process = Heavyweight
A program in execution. An independent unit with its own memory space, code, data, stack, registers, and PID. Created via fork() system call (kernel involvement). OS treats it as a completely separate entity.
🧵 Thread = Lightweight
A sub-unit within a process. Multiple threads of the same process share code, data, and heap memory. Each thread only has its own stack and registers. Created by application (no kernel system call needed for user-level threads).
11B

What a Process Contains vs What Threads Share

🏗️ Process Anatomy — Components of a Process
⚙️ One Process — Everything It Contains
📄
Code Program instructions (compiled bytecode)
🗄️
Data / Files Global variables, open file handles, heap memory
📚
Stack Function calls, local variables, return addresses
🔢
Registers CPU registers, program counter, I/O registers
🏷️
PID / PCB Process ID, Process Control Block (saved state)
When fork() creates a child process, ALL of these components are duplicated — a complete independent copy. This is why processes are "heavyweight."
🧵 Thread Structure — Shared vs Private Resources
⚙️ One Process (Contains 3 Threads)
🔗 SHARED BY ALL THREADS (same copy)
📄 Code — all threads run the same program code
🗄️ Data & Files — same global variables, same heap, same open files
🔒 PRIVATE — Each Thread Gets Its Own
Thread T1
📚 Stack (T1)
🔢 Registers (T1)
Thread T2
📚 Stack (T2)
🔢 Registers (T2)
Thread T3
📚 Stack (T3)
🔢 Registers (T3)
OS sees this entire box as ONE PROCESS. It doesn't distinguish T1, T2, T3 (for user-level threads).
Each thread has its own execution context (stack + registers) but shares everything else with other threads. This is why threads are "lightweight" — no duplication of code or data.
11C

When to Use Threads vs Processes

🖨️ Print Server Example — Thread vs Process

A print server receives print requests from 100 clients simultaneously. The same print code needs to run 100 times. Which approach is better?

❌ Using Processes (fork())
  • • 100 clients → 100 child processes created
  • • Each gets its OWN copy of print code
  • • Each gets its OWN copy of data
  • • 100 separate memory spaces
  • • HUGE overhead — memory, time, resources
  • • Context switching very slow
✅ Using Threads
  • • 100 clients → 100 threads within 1 process
  • • All share the SAME print code (no copy!)
  • • All share the SAME data space
  • • Only stack & registers differ per thread
  • • Minimal memory overhead
  • • Context switching very fast
Use threads when the same code needs to run multiple times in parallel (web servers, print servers, game engines). Use processes when tasks are truly independent and need full isolation.
11D

Context Switching — Process vs Thread

⏱️ Context Switching — Why Process Switching is Slower
🐢 Process Context Switch — SLOW
① Save P1's entire state to PCB
Save all registers, program counter, memory maps, open file handles, etc.
↓
② OS involvement required
Kernel manages PCB (Process Control Block). Uses OS memory and time.
↓
③ Load P2's entire state from its PCB
Restore all registers, reload memory maps, restore open files.
↓
④ Resume P2
Very expensive in time and OS resources.
⚡ Thread Context Switch — FAST
① Save T1's stack pointer + registers only
Code and data are shared — no need to save them. Minimal saving required.
↓
② No kernel involvement (user-level threads)
Application manages thread switching. No OS overhead. No mode switch.
↓
③ Load T2's stack pointer + registers
Restore minimal state. Memory is shared — already in place.
↓
④ Resume T2
Very fast — no heavy OS involvement needed.
Context switching speed: Thread >> Process. Because threads share memory — only registers and stack pointer need saving. Processes need their entire PCB saved/restored.
11E

★ Blocking Behaviour — Most Important Exam Point

⭐⭐ Star-marked exam point from lecture: The blocking behaviour of processes vs threads is critically different and appears frequently in GATE/UGC NET questions.
🚫 Blocking: Process vs Thread — Critical Difference
✅ Processes — INDEPENDENT
Process P1 makes an I/O request (e.g., read from disk). Kernel blocks P1.
P1 goes to Wait/Block state. OS scheduler picks next ready process.
P2, P3 continue running — completely unaffected! They are independent processes.
✅ Blocking ONE process does NOT block OTHER processes.
🖥️ P1 (BLOCKED)   🖥️ P2 (RUNNING)   🖥️ P3 (RUNNING)
❌ Threads — INTERDEPENDENT
Thread T1 makes an I/O request (e.g., read from disk). Request goes to kernel.
Kernel doesn't know about threads (for user-level threads). Kernel sees the WHOLE PROCESS. Kernel blocks the entire process.
T2 and T3 also get blocked — even though they had nothing to do with the I/O!
❌ Blocking ONE thread BLOCKS THE ENTIRE PROCESS (and all other threads in it).
🧵 T1 (I/O req) ➜ ⛔ Whole process blocked ➜ T2, T3 also stuck
Why does blocking T1 block T2 and T3? Because threading (at user level) is invisible to the OS kernel. The kernel sees one process. When that process makes an I/O request, the kernel blocks the entire process — it has no concept of "let the other threads of this process run." They share the same PID, the same scheduling unit.
★ KEY: Processes = independent (blocking one doesn't affect others). Threads = interdependent (blocking one blocks all in the same process).

🧵 Two types of threads (brief mention):
• User-Level Threads (ULT): Managed by application, not kernel. Fast to create/switch. But blocking one blocks all (kernel unaware).
• Kernel-Level Threads (KLT): Managed by OS kernel. Kernel aware of each thread. Blocking one doesn't block others. But creation/switching is slower (involves OS).
The lecture primarily discusses user-level threads.

Topic 12 — User Level Thread (ULT) vs Kernel Level Thread (KLT)

12A

Who Manages — User Library vs OS Kernel

🧵 ULT vs KLT — Who Creates and Manages Each
🧵 User Level Thread (ULT)
Managed by: User-level thread library (application's responsibility).
Example: pthread.h in C — functions like pthread_create(), pthread_join().
Creation: No system call needed — created via procedure/library calls only. Very fast.
Kernel awareness: Kernel has NO knowledge of ULTs. Kernel sees only the whole process as one unit.
OS role in execution: OS still needed for actual CPU scheduling, but creation and switching are entirely in user space.
⚙️ Kernel Level Thread (KLT)
Managed by: The OS kernel itself — kernel is directly responsible for creating and managing KLTs.
Creation: Requires system calls — OS involved. Slower than ULT creation. Somewhat similar to creating a process in terms of overhead.
Kernel awareness: Kernel fully knows each KLT individually. Kernel schedules each thread independently.
Examples: Linux kernel threads, Windows threads — OS manages them directly with kernel-side data structures.
Key: ULT = library manages, no system call, kernel unaware. KLT = OS kernel manages, system call required, kernel fully aware of each thread.

⏱️ Context Switching Speed — Fastest to Slowest
⚡⚡⚡
FASTEST
User Level Thread
No OS/kernel involved. Only save/restore stack pointer + registers. Shared code & data already in place. Pure user-space switch.
›
⚡⚡

12C

★ Blocking Behaviour — ULT vs KLT (Most Asked)

⭐⭐ GATE/UGC NET Star Point: Blocking behaviour is the most-asked ULT vs KLT difference in competitive exams.
🚫 Blocking: ULT vs KLT — Critical Difference
❌ ULT — One block = ALL blocked
Scenario: Process has T1, T2, T3 (user-level). T1 requests disk I/O.

  1. T1's I/O request reaches kernel.
  2. Kernel has NO knowledge of T1/T2/T3 as individual threads.
  3. Kernel sees ONE PROCESS making an I/O request.
  4. Kernel blocks the ENTIRE PROCESS.
  5. T2 and T3 also stuck — even though they wanted to run!
❌ One ULT block → whole process blocked (T1 + T2 + T3 all stuck)
✅ KLT — One blocks, others continue
Scenario: Process has T1, T2, T3 (kernel-level). T1 requests disk I/O.

  1. T1's I/O request reaches kernel.
  2. Kernel KNOWS T1, T2, T3 are separate threads (it created them!).
  3. Kernel blocks ONLY T1 — puts T1 in Wait state.
  4. Kernel scheduler picks T2 or T3 to run next.
  5. T2 and T3 continue executing normally.
✅ One KLT block → only that thread. T2 and T3 keep running.
Root cause: ULT — kernel blind to threads (sees one process → blocks everything). KLT — kernel sees each thread individually → blocks selectively.

Thread Mapping Models — Solving the ULT Blocking Problem

Since ULTs have the "one blocks all" problem, operating systems use mapping models that connect ULTs to KLTs. The mapping tells the kernel how many user threads correspond to how many kernel threads.

🗺️ Thread Mapping Models — Many-to-One, One-to-One, Many-to-Many
① Many-to-One (M:1)

KLT1 only
Multiple ULTs → Single KLT. All user threads multiplexed over one kernel thread. Only one ULT runs at a time.
⚠ Problem: One ULT blocks → all blocked. Cannot use multiple CPUs in parallel. Not for multiprocessors.
② One-to-One (1:1) — Used by Linux & Windows

Each ULT → Its own KLT. If one KLT blocks → others keep running (kernel sees them separately). True multiprocessor parallelism — different KLTs can run on different CPU cores.
⚠ Problem: Creating many threads = many kernel threads = overhead.
✅ Used by: Linux, Windows. Most common modern approach.
③ Many-to-Many (M:N) — Hybrid Model

Many ULTs → Fewer KLTs. Best of both worlds. Multiple ULTs multiplexed over multiple KLTs. OS can schedule another KLT when one blocks. True parallelism possible without 1:1 overhead.
✅ Benefit: No overhead of 1:1, no blocking-all problem of M:1. Flexible and efficient.
⚡ Also called: Hybrid model.
Linux uses One-to-One (1:1). Many-to-Many resolves both limitations (blocking + overhead). For exams: know which model solves the blocking problem (1:1 and M:N) and which doesn't (M:1).
12E

GATE 2014 — Code Sharing Clarification (Important!)

⭐ GATE 2014 asked: "Kernel level threads do NOT share the code section." — This statement is FALSE. Here is the correct explanation:
📋 GATE 2014 Clarification — Do KLTs Share Code?
❌ WRONG STATEMENT (GATE 2014 option):
"Kernel level threads do not share the code section."
✅ CORRECT:
Both ULT and KLT share code and data. They are still THREADS — the fundamental property of any thread is sharing code + data.

The "user" or "kernel" label only refers to WHO MANAGES the thread, not what it shares. Regardless of level:
• ✅ All threads share: Code section + Data section (global variables, heap)
• ❌ Each thread keeps private: Stack + Registers

That is why both ULT and KLT are called lightweight processes — no code/data duplication.
Universal thread rule: ANY thread (ULT or KLT) ALWAYS shares code + data, and ALWAYS has its own stack + registers. The type only affects: creation method, context switching speed, blocking behaviour, and kernel awareness.
This GATE 2014 trap catches students who think KLTs are "more process-like" and therefore don't share. But threads always share code/data regardless of their level.

Process vs Thread — Core Differences
Process
Heavyweight. Own code, data, stack, registers, PID, memory address space. Created by fork() (system call = kernel involved).
Thread
Lightweight. Shares code & data with other threads. Only has own stack and registers. Created by application (no system call for user-level).
OS view of Process
Each process = separate entity with its own PID. OS schedules and manages each independently.
OS view of Thread
OS treats the whole process as ONE unit (for user-level threads). Does NOT know individual threads exist.
What threads share
Code + Data + Heap + Open files. Same memory address space.
What threads keep private
Stack (own function call history) + Registers (own execution state). That's it.
Print server example
100 clients → use 100 threads (not 100 child processes). Threads share the same print code — no duplication overhead.

★ Context Switching & Blocking — Star Points
Process context switch
SLOW — must save/restore entire PCB (all registers, memory maps, file handles). Kernel (OS) involved. Mode switch required.
Thread context switch
FAST — only save/restore stack pointer + CPU registers. No OS involvement needed. Memory is shared — no reload.
Why time slicing?
Time quantum (e.g., 2ms) → CPU runs P1/T1 for 2ms → switches to P2/T2 → this switching is context switching.
Blocking a process
INDEPENDENT — blocking P1 (for I/O) does NOT block P2 or P3. Other processes continue running normally.
Blocking a thread
INTERDEPENDENT — blocking T1 blocks the ENTIRE process (T2 and T3 also blocked). Kernel sees one process → blocks whole thing.
Why thread blocking is total
Kernel (for user-level threads) doesn't know individual threads exist. When T1 makes I/O request → kernel blocks the whole process → T2, T3 also stuck.
Processes = independent
Using different memories & addresses → blocking one has zero effect on others.
Threads = interdependent
Sharing memory & addresses → blocking via kernel affects the shared process unit.

ULT vs KLT — Core Differences
ULT managed by
User-level thread library (application). e.g., pthread.h. Functions: pthread_create(), pthread_join(). No kernel involvement in creation.
KLT managed by
OS Kernel itself. Kernel creates and tracks each thread with kernel-side data structures.
ULT creation
No system call — library/procedure call. Very fast to create.
KLT creation
Requires system call — OS involved. Slower. Comparable overhead to creating a process.
Kernel awareness — ULT
Kernel has NO knowledge of individual ULTs. Sees the whole process as one unit.
Kernel awareness — KLT
Kernel fully knows each KLT individually. Schedules them independently.
Context switching speed
ULT switching > KLT switching > Process switching (fastest to slowest). ULT = pure user space. KLT = kernel invoked each time. Process = entire PCB saved/restored.
ULT blocking ⭐
One ULT blocks → ENTIRE PROCESS blocked. Kernel blind to threads → blocks the whole process when any I/O is requested.
KLT blocking ⭐
One KLT blocks → only that KLT blocked. Kernel aware of each KLT → can block one without affecting others. T2 and T3 continue.
GATE 2014 trap ⭐
"KLT does not share code" is WRONG (FALSE). Both ULT and KLT share code + data. All threads (ULT or KLT) always share code & data and always have own stack & registers. "User/kernel" only means WHO MANAGES — not what is shared.
Many-to-One (M:1)
Many ULTs → 1 KLT. Fast ULT switching but: one block = all blocked, no multiprocessor parallelism.
One-to-One (1:1)
Each ULT → own KLT. Solves blocking (only that KLT blocked) + enables true multiprocessor parallelism. Used by Linux and Windows.
Many-to-Many (M:N)
Many ULTs → fewer KLTs. Best of both worlds — no 1:1 overhead, no M:1 blocking problem. Hybrid model.
