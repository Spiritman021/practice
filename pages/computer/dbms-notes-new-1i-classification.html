DBMS Classification

DBMS types classified by model, users, distribution, licensing and purpose.

← PreviousQuizNext →

1. Why Classify DBMSs?

A database stores data in an organized way, while a DBMS manages it. DBMSs are classified because they differ in organization, processing, access, users, location, cost and purpose.

2. Classification by Data Model

The data model decides how data is organized, represented and accessed.

TypeCore ideaExample / use
RDBMSRows and columns; keys connect tables.MySQL for users/orders; also Oracle Database and Microsoft SQL Server.
OODBMSStores objects like Java/C++; handles nested, complex structures directly.db4o in embedded or complex inventory systems.
HierarchicalTree structure; one parent can have many children.IBM IMS for banking customer records.
NetworkWeb-like structure with multiple parent–child links.IDMS for many-to-many supply-chain relationships.
NoSQLFlexible schemas for large unstructured or semi-structured data.MongoDB for scalable analytics and content management.

NoSQL Categories

Document

Stores documents, often JSON. MongoDB.

Key–Value

Stores each value under a unique key. Redis.

Column-Family

Organizes related columns; useful for scale and analytics. Cassandra.

Graph

Optimized for complex relationships. Neo4j.

3. Classification by User Support

Single-User System

Supports one writer/user at a time and suits personal or small PC applications. SQLite permits many readers but normally one writer at a time.

Multi-User System

Allows simultaneous access for collaborative and enterprise work. PostgreSQL provides concurrency control and high availability for warehouses and complex transactions.

Exam caution: SQLite is often treated as single-user for writing even though concurrent reading is possible.

4. Classification by Database Distribution

Centralized DBMS

Stores data at one server or data center; access and processing are controlled centrally. Example: a university library’s central server.

Distributed DBMS (DDBMS)

Stores data across networked locations. Sites can process locally while remaining part of one larger database.

Example: Amazon DynamoDB is a distributed NoSQL service used across regions for highly available gaming or financial applications.

  • Homogeneous DDBMS: every site uses the same DBMS software.
  • Heterogeneous DDBMS: sites use different DBMS software; middleware integrates them.

5. Classification by Cost and Licensing

Open-Source DBMS

Free to use and modifiable; suitable for budget-conscious projects and small/medium businesses. Examples: MySQL, PostgreSQL.

Commercial DBMS

Paid licence with vendor support, extra features and service-level agreements. Oracle Database supports high-speed transactions and real-time analytics for large institutions.

6. Classification by Purpose

General-Purpose DBMS

Supports varied data, applications and use cases. SQL Server serves small businesses through large enterprises.

Special-Purpose DBMS

Optimized for one requirement, such as streaming data or specialized analytics. Telecom call-record systems need rapid ingestion and retrieval.

7. IBPS SO Quick Revision

  • RDBMS → tables and keys; MySQL, Oracle, SQL Server.
  • OODBMS → programming-language-like objects; db4o.
  • Hierarchical → tree/one parent; IMS.
  • Network → multiple parent–child links; IDMS.
  • NoSQL → flexible semi/unstructured data.
  • Document–MongoDB; key-value–Redis; column-family–Cassandra; graph–Neo4j.
  • Single-user example–SQLite; multi-user example–PostgreSQL.
  • Centralized stores at one location; distributed stores across networked sites.
  • Homogeneous uses the same DBMS; heterogeneous uses different DBMSs plus middleware.
  • Open-source examples–MySQL/PostgreSQL; commercial example–Oracle.
  • General-purpose handles varied work; special-purpose targets one workload.

Most likely questions: Tree model—hierarchical. Many parents—network. JSON documents—MongoDB. Key-value—Redis. Different DBMSs at sites—heterogeneous DDBMS.
