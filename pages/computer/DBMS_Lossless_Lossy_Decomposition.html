Lossless vs Lossy
Decomposition

When a normalized table is decomposed, joining the smaller tables back should not create spurious tuples. That is the idea of lossless decomposition.

Natural JoinSpurious Tuples Common AttributeCandidate Key Rule
1

Why Decomposition Matters

During normalization, a large table is decomposed into smaller tables. E. F. Codd's decomposition rules focus on two checks: lossless decomposition and dependency preservation.

This chapter focuses on the first check: after decomposing and joining back, the result should match the original table exactly.

Natural Join Result

Natural join matches rows where the common attribute has equal values. Here, common attribute B has duplicate value 2, so extra combinations appear.

Original table had 3 tuples. Join result has 5 tuples, so decomposition is lossy.
For the query “find C where A = 1”, the original table gives only C = 1, but the joined table gives C = 1 and C = 2. This inconsistency is called lossy decomposition.

Lossless Fix

The common attribute should be a candidate key or super key of at least one decomposed table. Here, keep A common because A is unique.
