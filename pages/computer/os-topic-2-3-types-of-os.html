Overview — All Types of OS

Operating systems are classified into several types based on how they handle jobs, processes, users, and timing. Here is the full category list:

🗂️ Types of Operating System — Category Overview
📦 Batch OS
Group similar jobs into batches; execute without user interaction. (1960s era)
📂 Multi-programmed OS
Multiple processes in RAM; CPU switches only when process goes for I/O (non-preemptive).
⏱️ Multi-tasking / Time-sharing OS
CPU switches between processes on a fixed time quantum (preemptive). Modern laptops use this.
⚡ Real-Time OS (RTOS)
Guarantees response within strict deadlines. Hard RTOS (airplanes, medical devices) and Soft RTOS.
🌐 Distributed OS
Multiple networked computers appear as one system. Resources shared across the network.
🔗 Clustered & Embedded OS
Clustered: multiple computers work on one task. Embedded: OS built into devices (microwave, washing machine).
The first three types (Batch, Multi-programmed, Multi-tasking) are the most exam-critical. Multi-programmed vs Multi-tasking difference is a guaranteed exam topic.
1

Batch Operating System 1960s Era

📦

Batch Operating System

1960s · IBM FORTRAN · IBSYS 709X · No Direct Interaction

Core Idea: Group (batch) similar jobs together and feed them to the computer for sequential execution — without any user interaction during execution.

🏛️ Historical Context — How Batch OS Worked in 1960s
1

Computers were huge and rare — only at big organisations

In the 1960s, only major companies like NASA, ISRO, or research labs needed computers. Ordinary people did not have personal computers. The machines were enormous and stored centrally in a company building.

2

Load your job onto Punch Cards / Paper Tape / Magnetic Tape

If you had a computation to do — say a set of mathematical calculations — you encoded your program offline onto a punch card (physical card with holes punched to represent data) or magnetic tape. The computer was not directly accessible to you.

3

Hand the punch card to the Operator

You physically went to the computing centre and handed your punch card to an Operator (a trained person who managed the computer system). You did NOT insert anything into the computer yourself. The operator was the intermediary.

4

Operator creates Batches of similar jobs

The operator collected all submitted jobs and grouped similar types of jobs into batches (e.g., Batch B1 has all calculation jobs, Batch B2 has all data processing jobs). Similar jobs are grouped because they need similar system resources — this reduces setup overhead between jobs.

5

CPU executes jobs one by one in the batch

The OS picks Job 1 from Batch 1 and gives it to the CPU. CPU executes it. If the job requests an I/O operation (reading from disk, accessing a device), the CPU sits idle waiting for the I/O to complete — then continues. This idle time was a major disadvantage.

6

Collect result after days, weeks, or months

You had no idea when your job would be executed — it depended on how many jobs were ahead of yours. Results were printed or recorded offline on a tape. You came back later to collect them. There was zero interaction once you submitted the job.

This entire process was the "Batch" workflow — submit, wait, collect. No user interaction possible once submitted.

⚠️ CPU Idle Problem: If Job1 requests an I/O operation (read from disk, access device) → CPU waits doing nothing until I/O completes → wasted CPU time is the biggest disadvantage of Batch OS.
Jobs in the same batch are of similar type, so no reconfiguration is needed between them — reducing setup overhead.
🖥️ Refinement — Monitors: To eliminate the need for human operators, early resident monitors were introduced. ✅ Advantages
  • CPU is kept busy (within a batch, no idle time between similar jobs)
  • Operator can group similar jobs — reduces setup overhead
  • Good for repetitive, large-volume computing tasks
  • Easy to manage from operator's side
❌ Disadvantages
  • CPU idle time when a job waits for I/O — huge waste
  • No user interaction once job is submitted
  • Result may take days, weeks, or even months
  • Debugging is very difficult — no real-time feedback
  • Non-preemptive — CPU cannot switch to another job during I/O wait
📝 Exam note: IBM FORTRAN Monitor System (FMS) and IBSYS 709X are the key systems from the Batch OS era. If a question asks about Batch OS systems/examples — these two are the answer. Both are from the 1960s.

Multi-Programmed Operating System Non-Preemptive

📂

Multi-Programmed OS

Multiple Processes in RAM · Non-Preemptive · CPU switches only on voluntary I/O request

Core Idea: Load multiple processes simultaneously into RAM. When the currently running process voluntarily goes for an I/O operation, the OS immediately gives the CPU to another process — so the CPU is never idle waiting for I/O.

🎯 Key Goal: Maximise CPU utilisation — keep the CPU busy as much as possible. Reduce idle time.

The key insight: CPU never sits idle. When one process is doing I/O, another process uses the CPU.
🎓 Student Analogy — Multi-programmed OS

Imagine: 10 students in a classroom, each has 5 questions to solve. I am the CPU. In multi-programmed mode:

Step 1
I go to Student 1 and solve all 5 questions — Q1, Q2, Q3, Q4, Q5.
Step 2
ONLY if Student 1 says "I need to go do something else (I/O)" mid-way, I switch. Otherwise, all 5 questions first.
Step 3
Then I go to Student 2 and solve all their 5 questions. Then Student 3, 4... all the way to Student 10.
⚠️ Problem: Student 10 has to wait for ALL 9 students before getting any attention — very long response time for later processes.
Non-preemptive: the CPU does NOT switch voluntarily. The process must give up the CPU itself (voluntary I/O request).
📊 CPU Execution Timeline — Multi-programmed (Non-preemptive)
Timeline →
P1 Runs fully until I/O or completion
IDLE P1 I/O wait
P2 Runs fully until I/O or completion
P3 Runs completely
✅ CPU Utilisation improved compared to Batch OS — when P1 does I/O, P2 runs instead of CPU sitting idle.
⚠️ Response time is poor — P3 must wait for all of P1 and P2 to finish first.
CPU only switches when the currently running process voluntarily releases it (for I/O or completion).
✅ Advantages
  • CPU utilisation is high — CPU rarely sits idle
  • Multiple processes in RAM ready to use CPU instantly
  • Better than Batch OS — I/O waiting doesn't waste CPU
  • Throughput improves significantly over Batch OS
❌ Disadvantages
  • Poor response time — later processes wait for earlier ones to fully complete
  • Non-preemptive — CPU cannot be taken away from a running process forcefully
  • One long-running process can monopolise the CPU
  • Not suitable for interactive (real-time user input) applications
3

Multi-Tasking / Time-Sharing OS Preemptive Modern Laptops

⏱️

Multi-Tasking / Time-Sharing OS

Preemptive · Fixed Time Quantum · Better Response Time · Round Robin Algorithm

Core Idea: The CPU is given to each process for a fixed time interval (time quantum). When the time is up, the CPU is forcefully taken away (preempted) and given to the next process — regardless of whether the current process has finished.

🎯 Key Goals: (1) Minimise CPU idle time — same as multi-programmed. (2) Improve Response Time — the additional and more important advantage over multi-programmed OS.
⏱️ Multi-tasking OS — Time Quantum Concept
Time Quantum (Q)
e.g. 2 ms
A time quantum (Q) is a fixed time slice decided in advance. Each process gets the CPU for exactly Q time units. If it finishes in less — great. If not — it is preempted and put back in the ready queue.

🔧 Algorithm used: Round Robin (the natural implementation of time-sharing)
Each process gets CPU for exactly one time quantum, then goes to the back of the ready queue. This creates the illusion of parallel execution.
⏱️ Response Time: Multi-programmed vs Multi-tasking
Multi-programmed
P1 fully → P2 fully → P3 fully → P4 finally gets response.

Response Time for P4 = burst time of P1 + P2 + P3
(Could be very long if P1, P2, P3 are heavy processes)
Multi-tasking
P1(Q) → P2(Q) → P3(Q) → P4 gets CPU in Round 1!

Response Time for P4 = 3 × time quantum
(Much smaller — proportional only to quantum × number of processes before it)
This is WHY modern OS use time-sharing — interactive applications (typing, browsing) require fast response times.
✅ Advantages
  • Low idle time — CPU stays busy across all processes
  • Better response time — all processes get CPU quickly
  • Fair CPU distribution — no single process can monopolise
  • Suitable for interactive systems (typing, GUI, browsing)
  • This is what your laptop/desktop OS uses right now
❌ Disadvantages
  • Context switching overhead — switching between processes takes time
  • If time quantum is too small — most time spent switching, not executing
  • If time quantum is too large — behaves like multi-programmed (poor response)
  • Choosing the right Q value is a design challenge
🖥️ Real-world usage: The operating system on your laptop — Windows, macOS, Linux — uses a multi-tasking (time-sharing) approach. When you have Word, Chrome, and Spotify all "running" at the same time, they are actually taking turns using the CPU. The switching is so fast (milliseconds) that it feels simultaneous. The primary algorithm used is Round Robin.

★ Multi-programmed vs Multi-tasking is the most exam-tested distinction in all of OS Types.
⭐ The ONE key difference to always remember: Multi-programmed focuses on CPU Utilisation (keep CPU busy). Multi-tasking adds Response Time on top of that. Multi-tasking is preemptive; multi-programmed is non-preemptive. Multi-tasking is what modern computers use.
1
Batch Operating System
Core idea
Group similar jobs into batches; submit to operator; CPU executes batch sequentially without user interaction.
Era
1960s. Only large organisations (NASA, ISRO) had computers. Personal computers did not exist.
Job submission
Offline onto Punch Cards / Paper Tape / Magnetic Tape → hand to Operator → Operator loads batch → CPU executes.
CPU behaviour
Non-preemptive. Executes one job at a time. If job requests I/O → CPU sits IDLE waiting (biggest disadvantage).
User interaction
ZERO. Once submitted, user has no control. Result may arrive after days, weeks, or months.
Refinement
Monitors introduced to eliminate need for operators (resident monitor reads punch cards directly).
Key systems (exam)
IBM FORTRAN Monitor System (FMS) and IBSYS 709X — both from 1960s.
Disadvantages
High CPU idle time (I/O wait), no user interaction, unknown wait time, poor debugging support.
2
Multi-programmed OS
Core idea
Load multiple processes simultaneously into RAM. When process goes to I/O, CPU immediately runs another process.
Preemption type
NON-PREEMPTIVE — CPU can only be taken from a process if the process voluntarily requests I/O or terminates.
Switching trigger
Process voluntarily goes for I/O operation (e.g., read from disk, access device). NOT on a timer.
CPU idle time
Reduced compared to Batch OS. When P1 does I/O, P2 uses the CPU — no wasted time.
Response time
Poor. Process P10 must wait for P1–P9 to fully complete or do I/O before getting any CPU time.
Goal
Maximise CPU utilisation — keep CPU as busy as possible.
Student analogy
Solve ALL 5 questions for Student 1, then ALL 5 for Student 2… Student 10 waits for everyone before getting any help.
3
Multi-tasking / Time-sharing OS
Core idea
CPU given to each process for a fixed TIME QUANTUM. After quantum expires, CPU forcefully switches (preempts) to next process.
Preemption type
PREEMPTIVE — OS forces the switch when time quantum expires. Process does not need to give up CPU voluntarily.
Switching trigger
Time quantum expires OR process requests I/O (whichever comes first).
CPU idle time
Very low. CPU always has processes to execute.
Response time
Good / Low. All processes get CPU within 1 round. P10 gets response after only 9 × quantum wait.
Goals
CPU utilisation AND better response time (both are goals, unlike multi-programmed which only focuses on CPU use).
Algorithm used
Round Robin — each process gets equal time quantum in circular fashion.
Real-world usage
Your laptop/desktop right now. Windows, macOS, Linux all use time-sharing. Enables running Word + Chrome + Music simultaneously.
Student analogy
Solve 2 questions for each student in a round → back to Student 1 for next 2 → everyone gets progress in every round.
