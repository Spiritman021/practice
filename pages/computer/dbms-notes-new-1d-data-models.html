DBMS Data Models

Flat, Entity–Relationship and Relational models—short and exam-focused.

← PreviousQuizNext →

1. What is a Data Model?

A data model defines the logical structure of a database: how data is connected, processed and stored.

Data models introduce abstraction—users can understand the database logically without knowing every physical storage detail.

Memory line: Data model = rules and concepts used to organize data and relationships.

2. Flat Data Model

The earliest model kept all data on the same plane, usually in one flat file or table.

  • Simple to create and understand.
  • Not scientifically structured.
  • Repeats the same facts, causing duplication.
  • Repeated data produces update anomalies: one copy may be changed while another remains old.

Example: If every student-course row repeats the student's name and class, one name correction must be made in many rows.

3. Entity–Relationship (ER) Model

The ER Model represents a real-world situation using entities, attributes, relationships and constraints. It is best suited to the conceptual design stage—before tables are implemented.

Entity

A distinguishable real-world object. Example: Student.

Entity Set

A collection of similar entities, such as all students.

Attribute

A property of an entity: Name, Age or Class.

Domain

The allowed set of values for an attribute. Age may allow valid integer ages.

Relationship

A logical association among entities. Example: Student enrolls in Course.

Constraints

Rules restricting valid data or associations.

4. Mapping Cardinalities

Mapping cardinality tells how many entities of one set can be associated with entities of another set.

TypeMeaningExample
One-to-One (1:1)One A relates to one B.Person ↔ Passport
One-to-Many (1:N)One A relates to many B.Department → Employees
Many-to-One (N:1)Many A relate to one B.Employees → Department
Many-to-Many (M:N)Many A relate to many B.Students ↔ Courses

Exam trap: 1:N and N:1 describe the same association from opposite directions. Always check which entity is mentioned first.

5. Relational Model

The Relational Model is the most popular DBMS data model. It is more scientific and is based on first-order predicate logic. A table is mathematically treated as an n-ary relation.

n-ary means the relation has n attributes (columns). A table with four columns is a 4-ary relation.

Main properties

  • Data is stored in tables called relations.
  • Relations can be normalized to reduce redundancy and anomalies.
  • In normalized relations, stored values are atomic—one indivisible value per cell.
  • Each row is uniquely identifiable, normally through a key.
  • All values in a column come from the same domain.

Easy example: A STUDENT relation stores one student per row; StudentID uniquely identifies the row; Age contains only values from the permitted age domain.

6. Quick Comparison

ModelMain ideaBest use / issue
FlatAll data in one planeSimple, but duplicates data and causes anomalies

7. IBPS SO Quick Revision

  • Data models define logical structure, connections, processing and storage.
  • They provide abstraction in DBMS.
  • Flat models cause duplication and update anomalies.
  • ER Model is best for conceptual design.
  • Entity = real-world object; attribute = property; domain = allowed values.
  • Relationship = logical association between entities.
  • Cardinalities: 1:1, 1:N, N:1 and M:N.
  • Relational Model is most popular and based on first-order predicate logic.
  • Relation = table; tuple = row; attribute = column.
  • Normalized values are atomic; rows are unique; each column has one domain.

Most likely questions: Best model for conceptual design—ER Model. Most popular model—Relational Model. Repeated flat-file data causes—redundancy and update anomalies. Allowed values of an attribute—domain.
