Codd’s 12 Rules

Requirements proposed by Dr. Edgar F. Codd for a true relational DBMS.

← PreviousQuizNext →

Absolutely. For IBPS SO IT Officer, you do not need to memorize the exact wording of Codd's Rules. Instead, understand what each rule means, why it exists, and remember one simple example. Many questions are direct, such as:

  • Which rule states that data should be stored in tables?
  • Which rule is related to NULL values?
  • Which rule talks about data independence?
  • Which rule is about distributed databases?

Let's understand all the rules in the easiest possible way.

Codd's 12 Rules (IBPS SO IT)

Who was Codd?

Dr. Edgar F. Codd is called the Father of Relational Database Management System (RDBMS).

He proposed 12 rules (actually 13 including Rule 0) to decide whether a database system is truly a Relational Database Management System (RDBMS).

Think of these rules as:

"If a database follows these rules, then it is a proper RDBMS."

Rule 0 (Foundation Rule)

Although the chapter starts from Rule 1, there is actually a Rule 0.

Rule

A database should manage data only through relational capabilities.

In simple words,

Everything should work according to relational database principles.

If a database uses some non-relational tricks to manage data, then it is not a pure RDBMS.

Rule 1 : Information Rule

Statement

Everything in the database must be stored in the form of tables.

This includes:

  • User data
  • Metadata (information about the database)

Simple Meaning

No matter what information you have,

it should be stored inside rows and columns.

Everything = Table

Example

Student Table

Roll NoNameMarks
101Rahul90
102Aman85

Database information (metadata)

Table NameColumns
StudentRoll, Name, Marks

Even metadata is stored in tables.

Easy Memory Trick

Everything = Table

IBPS Question

Which rule says everything should be stored in tables?

✅ Information Rule

Rule 2 : Guaranteed Access Rule

Statement

Every data item must be accessible using:

  • Table Name
  • Primary Key
  • Column Name

No pointers or hidden methods should be needed.

Simple Meaning

Every value should have one unique address.

Just like your house has:

  • City
  • Street
  • House Number

Similarly, database values have:

  • Table
  • Row (Primary Key)
  • Column

Example

Student Table

RollNameMarks
101Rahul90

To get Rahul's marks:

Table = Student

Primary Key = 101

Column = Marks

Output = 90

You never need memory addresses or pointers.

Easy Memory Trick

Every value has an address.

(Table + PK + Column)

Rule 3 : Systematic Treatment of NULL Values

Statement

NULL values must be handled consistently.

What is NULL?

NULL means:

Not Zero

Not Blank

Not Empty String

It means No Value.

NULL may represent:

1. Data Missing

We forgot to enter it.

Example

Phone Number = NULL

2. Data Unknown

The value exists but we don't know it.

Example

Blood Group = NULL

3. Not Applicable

There is no value.

Example

Spouse Name of an unmarried person

NULL

Why Important?

The database should treat every NULL the same way.

Easy Memory Trick

NULL = No Value

IBPS Point

NULL ≠ 0

NULL ≠ Blank

Rule 4 : Active Online Catalog Rule

Statement

Database structure should also be stored inside the database.

This structure is called

Data Dictionary (Catalog).

What is Data Dictionary?

It stores:

  • Table names
  • Column names
  • Constraints
  • Data types
  • Indexes

Basically,

Database information about the database.

Example

Suppose you ask

"What columns exist in Student table?"

Database should answer this using SQL.

Example

DESCRIBE Student;

or

SHOW TABLES;

The catalog itself is queryable.

Easy Memory Trick

Database stores information about itself.

Rule 5 : Comprehensive Data Sublanguage Rule

Statement

There should be one language that can perform all database operations.

That language should support

  • Creating tables
  • Updating records
  • Deleting records
  • Transactions
  • Security
  • Constraints

That language is

SQL

SQL includes

DDL

CREATE
ALTER
DROP

DML

INSERT
UPDATE
DELETE
SELECT

Transaction Commands

COMMIT
ROLLBACK
SAVEPOINT

Meaning

You should not need another language.

Everything should be possible through SQL.

Easy Memory Trick

One language does everything.

Rule 6 : View Updating Rule

First Understand View

A View is a virtual table.

Example

Original Student Table

RollNameMarks

View

NameMarks
Rahul90

If a view can theoretically be updated,

the database should allow it.

Example

Suppose view contains only

Name

Marks

Updating Marks

UPDATE StudentView
SET Marks=95;

Should work.

Note

Not every view is updateable.

Only updateable views should be updateable.

Easy Memory Trick

Update the View if possible.

Rule 7 : High-Level Insert, Update and Delete Rule

Statement

Database should support operations on

Entire set of rows,

not only one row.

Instead of

Updating row by row,

it should update many rows together.

Example

Instead of

UPDATE Student
SET Marks=90
WHERE Roll=1;

It should also allow

UPDATE Student
SET Marks=Marks+5;

Every student gets +5 marks.

Similarly

Insert multiple rows.

Delete multiple rows.

Also Supports

  • UNION
  • INTERSECTION
  • MINUS (EXCEPT)

These are set operations.

Easy Memory Trick

Work on sets, not just single rows.

Rule 8 : Physical Data Independence

Statement

Changing physical storage should not affect applications.

Physical changes include

  • Moving files
  • Changing disk
  • Creating index
  • Compression

Example

Today

Database stored on HDD.

Tomorrow

Stored on SSD.

Application keeps working.

Easy Memory Trick

Storage changes → No effect

Rule 9 : Logical Data Independence

Statement

Changing logical structure should not affect applications.

Logical changes include

  • Split table
  • Merge table
  • Add columns
  • Remove columns (carefully)

Example

Earlier

Student Table

Later

Student

Result

StudentAddress

Application should still work.

Why Difficult?

Applications usually depend on table structure.

Making them independent is hard.

Easy Memory Trick

Table structure changes → App still works.

Rule 10 : Integrity Independence

What is Integrity?

Rules that maintain correct data.

Examples

Age > 18

Salary > 0

Marks ≤ 100

Primary Key

Foreign Key

Unique

Check Constraint

Rule

These constraints should stay inside the database,

not inside application code.

Example

Instead of Java checking

if(age<18)

Database should enforce

CHECK(age>=18)

Easy Memory Trick

Database protects data.

Rule 11 : Distribution Independence

Statement

User should not know where data is stored.

Suppose

Customer Table

Delhi Server

Order Table

Mumbai Server

User writes

SELECT *
FROM Customer;

User should never know

where data is physically located.

Easy Memory Trick

Many locations

Looks like one database.

Very Important

This rule is the basis of

Distributed Databases.

Rule 12 : Non-Subversion Rule

Statement

No low-level access should bypass database rules.

Suppose

Someone directly accesses storage files.

He should still NOT be able to bypass

  • Constraints
  • Security
  • Integrity

Example

Even administrator tools should obey

Primary Key

Foreign Key

Permissions

Easy Memory Trick

No shortcut around security.

IBPS SO IT Exam Tips

The following rules are asked most frequently in objective exams:

  • Rule 1 (Information Rule): Everything is stored in tables.
  • Rule 2 (Guaranteed Access): Access every value using Table Name + Primary Key + Column Name.
  • Rule 3 (NULL): NULL means no value (missing, unknown, or not applicable), not zero or blank.
  • Rule 5 (Comprehensive Data Sublanguage): SQL should handle all database operations.
  • Rule 8 (Physical Data Independence): Changes in physical storage should not affect applications.
  • Rule 9 (Logical Data Independence): Changes in the logical schema should not affect applications (harder to achieve than physical independence).
  • Rule 10 (Integrity Independence): Constraints should be enforced by the database, not the application.
  • Rule 11 (Distribution Independence): Users should not know where data is physically stored; this underpins distributed databases.

One-Line Memory Story

Imagine an ideal RDBMS:

Everything is stored in tables (Rule 1), every value has a unique address (Rule 2), NULL is treated consistently (Rule 3), the database stores information about itself (Rule 4), SQL does everything (Rule 5), updateable views can be updated (Rule 6), operations work on entire sets of rows (Rule 7), storage changes don't break apps (Rule 8), schema changes don't break apps (Rule 9), the database enforces constraints (Rule 10), distributed data looks like one database (Rule 11), and nobody can bypass its security or integrity rules (Rule 12).

Remembering this story helps you recall the rules in order during the IBPS SO IT exam.
