1. Introduction: What is an Operating System?

An Operating System (OS) is system software that acts as an intermediary between the user/applications and computer hardware. It manages resources and provides an environment in which programs can run.

There is no single universally precise definition because the emphasis changes with context. For exams, remember that an OS is a program/system software with three fundamental roles.

Interface analogy: A fan regulator, car steering wheel, or microwave buttons let an ordinary user control a complex machine conveniently. Similarly, an OS hides low-level hardware complexity and offers an easy way to use a computer.

Without an OS, directly controlling hardware may be technically possible in limited cases, but it would require detailed knowledge of circuits, device operations, memory, and processor instructions.

2. Three Fundamental Roles of an OS

1. IntermediaryReceives requests from users/applications, gets work done by hardware, and returns the result.
2. Resource ManagerControls and allocates hardware and software resources among competing programs.
3. PlatformProvides the installation and execution environment for application programs.

OS as Resource Manager / Allocator

The OS controls and coordinates resource usage among different application programs in a fair, controlled, and efficient manner.

  • Hardware resources: CPU time, main memory, secondary storage, registers, input/output devices, and system buses (data, address, and control buses).
  • Software/logical resources: files, access permissions, authorization mechanisms, synchronization tools such as semaphores, and shared services.
  • Example: After a user opens a movie, the OS loads the program, assigns CPU and memory, reads data from storage, controls the display and speakers, and coordinates all these activities during execution.
Best exam definition: “An OS is system software that acts as an intermediary between user and hardware, manages and allocates system resources, and provides a platform and execution environment for application programs.”

3. Why Do We Need an Operating System?

The central purpose is to enable everyone to use hardware in a convenient and efficient manner.

  • It hides complicated hardware details from ordinary users.
  • It translates user/application requests into operations understood by hardware.
  • It coordinates multiple programs so they do not interfere with one another.
  • It shares limited resources according to defined policies and algorithms.
  • It offers common services so every application does not need to control hardware from scratch.
Government of India analogy: India has shared resources such as land, water, rivers, minerals, and people. The government regulates who may use them, how much may be used, and under what rules. In the same way, hardware belongs to the whole computer system, while the OS authorizes, allocates, and coordinates its use among programs. The analogy explains management—not that a government is literally an OS.

4. Abstract View of a Computer System

User→System & Application Programs→Operating System→Computer Hardware
  1. Computer hardware (Level 0): CPU, memory, registers, I/O devices, storage, and system buses provide basic computing resources.
  2. Operating System (Level 1): Controls and coordinates hardware use among programs.
  3. System and application programs (Level 2): Compilers, assemblers, text editors, database systems, browsers, games, etc. define how resources solve user problems.
  4. Users (Level 3): People or other systems that use applications to request work.

Typical request path: User asks an application to perform a task → application requests OS services → OS controls the hardware → result returns through the OS and application to the user.

6. Goals of an Operating System

A goal is the desired destination; functions are the activities performed to achieve it.

Primary GoalConvenience and user-friendliness: make the computer easy to use.
Secondary GoalEfficiency: use CPU, memory, storage, and devices effectively.
Other Secondary GoalsReliability, security, and maintainability/ease of evolution.
DSLR vs smartphone: A professional camera may offer better control and image quality, but many users prefer a phone because it is easier and faster to use. This illustrates why convenience is treated as the primary goal; efficiency or power alone is not enough if a system is impractical for its intended users.

7. Major Functions / Management Areas

Like ministries that carry out a government’s goals, the OS uses multiple management components to achieve convenience, efficiency, and reliability:

  • Process management: Creates, schedules, synchronizes, and terminates processes; allocates CPU time. This is a major part of OS study.
  • Memory management: Tracks memory usage and allocates/deallocates main memory.
  • File management: Organizes files and directories and controls their operations and permissions.
  • I/O and device management: Coordinates devices through drivers, buffering, caching, and related mechanisms.
  • Storage management: Manages primary and secondary storage, free space, and disk access.
  • Security and protection: Authentication, authorization, isolation, and controlled resource access.
  • Networking and communication: Supports communication between processes and connected systems.
  • Error detection and accounting: Detects failures, records resource usage, and helps monitor performance.

9. Multiprogramming Operating System

Multiprogramming means keeping more than one program/job in main memory at the same time so that, when the running job must wait, the OS can give the CPU to another ready job.

The central objective is high CPU utilization: the processor should not remain idle merely because one program is waiting for an I/O operation or another event. In the teacher’s words, “the show must go on” and “the processor waits for no one.”

Memory and execution model

  • Jobs first reside on secondary storage in a job pool, waiting to be admitted to main memory.
  • The CPU normally executes instructions of programs that have been loaded into main memory; it does not directly execute a job from slow secondary storage.
  • The OS and several jobs—such as A, B, and C—are kept in main memory simultaneously.
  • At any instant on a single CPU, only one job is actually executing. The other loaded jobs may be ready or waiting.
Jobs on disk→Job pool→Several jobs in RAM→One job on CPU

How it works

  1. The OS selects a ready job in main memory and gives it the CPU.
  2. The job runs until it finishes or has to wait—for example, for keyboard input, disk access, output, a child process, or another event.
  3. In a non-multiprogrammed system, the CPU could sit idle during this wait.
  4. In a multiprogrammed system, the OS switches the CPU to another ready job.
  5. When the first job’s wait finishes, it becomes ready again and can later regain the CPU.
  6. This continues as long as at least one job is ready to execute.
Critical misconception: Multiprogramming does not mean multiple programs execute at exactly the same instant on one CPU. It means multiple programs are available in memory, giving the CPU alternatives. True simultaneous execution requires multiple processing units/cores and is associated with multiprocessing.

Advantages

  • Higher and more efficient CPU utilization by overlapping one job’s I/O wait with another job’s CPU work.
  • Improved throughput because more work can be completed over a period of time.
  • Several processes share CPU time, making systems with multiple concurrent tasks practical.
  • The slide lists reduced response, waiting, and turnaround time as benefits; practically these may improve on average, but are not guaranteed for every individual job because jobs compete for resources.

Disadvantages / requirements

  • CPU scheduling is more complicated because the OS must select among ready jobs.
  • Memory management and protection become more complex because several jobs coexist in RAM.
  • The system needs job scheduling, process state tracking, context switching, resource protection, and mechanisms to prevent one job from corrupting another.

GATE 2002 / NET Dec 2012 question

Which features characterize a multiprogrammed OS?

  1. More than one program may be loaded into main memory at the same time.
  2. If a program waits for an event, another program is immediately scheduled.
  3. If a program terminates, another program is immediately scheduled.
Reasoning: (a) is the defining memory condition, and (b) captures the defining CPU-utilization behavior. Statement (c) can also occur in a uniprogrammed/batch system, so it does not uniquely identify multiprogramming. Conceptually, (a) and (b) are the distinguishing features (Option B). The teacher notes that many older published solutions accept all three (Option D) by treating (c) as a valid general behavior of such a system; hence the old question is ambiguous.

10. Multitasking and Time-Sharing

Time-sharing (or multitasking) is a logical extension of multiprogramming in which the CPU switches rapidly among ready tasks/users, giving each a small share of CPU time and creating the illusion of simultaneous execution.

Modern systems appear to play music, edit a document, and browse the web at the same time. On a single CPU/core, this apparent parallelism is achieved by rapid context switching.

Two requirements

  1. Multiprogramming: multiple programs must be present and ready for execution.
  2. Time-sharing policy: the CPU is divided into small time slices and shared fairly, commonly through preemptive scheduling such as Round Robin.
Street-food vendor analogy: A vendor has several customers. Instead of completing one customer’s entire order before touching the next, the vendor gives each order attention in turns. Every customer sees progress and nobody waits for one long order to finish. The CPU similarly rotates among ready tasks.

Precision note: Do not memorize “multiprogramming = one user” as an absolute definition. Its defining feature is multiple resident jobs and switching when the running job cannot proceed. Time-sharing adds frequent/preemptive sharing and interactive responsiveness, often across many users.

11. Multiprocessing OS / Tightly Coupled System

A multiprocessing system contains two or more processors/cores within one computer system, allowing multiple processes or threads to execute truly in parallel.

Unlike multiprogramming on a single CPU—where only one process executes at an instant—multiprocessing provides multiple execution units. The OS distributes runnable work among them to increase processing capacity and reliability.

Tightly coupled architecture: processors have private caches but share main memory and I/O through a common interconnect.

Why use multiprocessing?

  • True parallelism: Different processors can execute different processes/threads at the same instant.
  • Increased throughput: More work can be completed in less time.
  • Economy of scale: Processors share memory, storage, I/O, and other system facilities instead of requiring separate computers.
  • Improved reliability: In a properly designed system, failure of one processor may allow degraded service to continue on the remaining processors (graceful degradation).
  • Better energy use: Modern devices can activate more cores under heavy loads such as gaming or 4K video and idle/disable cores under light loads.
Do not confuse: Multiprogramming = many programs in memory with one CPU switching among them. Multiprocessing = multiple processors/cores, so actual parallel execution is possible.

12. Real-Time Operating System (RTOS)

An RTOS is designed to process events and complete tasks within specified timing constraints, producing predictable and deterministic responses.

In an ordinary system, completing a task correctly is important. In a real-time system, the result must be both logically correct and delivered on time. A correct result produced after its deadline may have little value—or may cause failure.

Mechanism and characteristics

  • Uses real-time scheduling to prioritize tasks according to urgency, importance, and deadlines.
  • May preempt a lower-priority task when a more urgent task becomes ready.
  • Seeks bounded and predictable response/interrupt latency, not merely a high average speed.
  • Emphasizes predictability, reliability, controlled resource use, and minimal latency.

Applications: aircraft and weapon control, air-traffic control, intensive-care equipment, industrial control, automotive airbags and anti-lock brakes, engine controllers, embedded devices, multimedia, and online processing.

Hard vs Soft Real-Time Systems

Usefulness over time: a hard real-time result loses its value abruptly at the deadline; a soft real-time result remains useful for a while but its value gradually declines.

Hard real-time example: If a defensive action is required before an aircraft crosses a boundary, executing it five seconds late may be useless or even harmful. Soft real-time example: A small delay or dropped frame during gaming or a video call reduces quality, but the application remains useful.
Important correction: “Real-time” does not mean “instantaneous” or necessarily fast. It means the system provides a response within a known, required time bound.

13. User and Operating-System Interface

A user interface is the mechanism through which a person gives instructions to the operating system and receives results. Common forms are CLI, GUI, and touchscreen interfaces.

The OS stands between users/applications and hardware, but the user still needs a convenient language for expressing requests. A command, mouse click, icon tap, or gesture ultimately represents an operation that software translates into lower-level requests.

Different front-end interfaces express requests differently, but all ultimately rely on operating-system services.

14. System Calls

A system call is the controlled mechanism by which a user program requests a service from the operating-system kernel—especially an operation the program cannot safely or directly perform itself.

Applications normally execute in user mode, with restricted privileges. The kernel executes in kernel mode, where it may control memory, devices, processes, files, and protected CPU operations. A system call crosses this protection boundary in a defined and validated way.

A system call changes execution from user mode to kernel mode; after the service completes, control and a result/status return to the application.

Security analogy: A protected person does not need personal control over every restricted weapon; authorized security personnel perform the protected service on their behalf. Likewise, applications do not receive unrestricted hardware control. They request the kernel to perform validated operations.

API, library function, and system call

  • An API specifies functions available to programmers, their parameters, expected behavior, and return values.
  • A library function is code invoked by the program. It may perform the work entirely in user space or act as a wrapper around one or more system calls.
  • A system call enters the kernel to request a protected OS service.
  • The caller needs to understand the API contract, not the kernel’s internal implementation.

Common API families include the Windows API, POSIX API for UNIX/Linux and related systems, and Java APIs used by programs running on the JVM.

Precision note: API calls and system calls are not always one-to-one. One API call may use zero, one, or multiple system calls, and the exact system-call mapping can differ by OS and implementation.

Example: printf() and write()

printf() is a C library function, not itself a system call. When output must be sent, its implementation may eventually invoke the OS’s write()-like service.

Mode Bit and Dual-Mode Operation

To protect the operating system and hardware, the CPU must distinguish between ordinary user code and trusted kernel code. Hardware therefore supports at least two execution modes:

  • User mode: Used while a user application is executing. Privileged instructions and direct access to protected resources are restricted.
  • Kernel mode: Also called supervisor, system, or privileged mode. The OS kernel may execute privileged instructions and manage hardware.
  • A hardware mode bit records the current mode. In the convention used by the slide, 0 = kernel mode and 1 = user mode.

System-call transition: the trap switches the CPU from user mode (1) to kernel mode (0); the return-from-system-call instruction restores user mode (1).

  1. The application executes normally in user mode, with mode bit 1.
  2. It requests an OS service by invoking a system call.
  3. A controlled trap transfers execution to a predefined kernel entry point and changes the mode bit to 0.
  4. The kernel validates the request and performs the privileged service.
  5. A return-from-system-call operation restores the application state, changes the mode bit to 1, and resumes user code.
Why this matters: The mode bit prevents a faulty or malicious user program from executing privileged instructions directly. The exact numeric encoding is architecture-dependent, but exam questions commonly use kernel = 0, user = 1.

Abstraction Across Software and Hardware Layers

Abstraction means hiding unnecessary implementation details and exposing only the operations required by the layer above.

A programmer should not need to know the electrical details of a processor to display text. Likewise, an application normally does not need to know how the kernel implements file access internally. Each layer provides a simpler interface while using the services of the layer below it.

Each layer depends on the contract exposed below it while hiding lower-level implementation details from the layer above.

Farmer analogy: A farmer uses land and chooses to grow sugarcane. A factory buys the sugarcane and decides whether to make sugar, jaggery, or another product. A restaurant buys sugar and decides whether to use it in tea, rasgulla, or gulab jamun. Each participant works at a different level and needs only the input/output contract of the adjacent level—not every internal detail of the full chain.

15. Preemptive and Non-Preemptive Scheduling

Source: GeeksforGeeks — Preemptive and Non-Preemptive Scheduling (accessed July 2026).

CPU scheduling is the OS activity of selecting a process from the ready queue and assigning the CPU to it. Scheduling policies aim to use the processor efficiently while controlling waiting, response, and turnaround times.

Preemptive scheduling

In preemptive scheduling, the OS may forcibly interrupt a running process and move it from the Running state back to the Ready state before its CPU burst finishes. The scheduler then assigns the CPU to another ready process.

Typical reasons for preemption:

  • The running process’s time quantum expires.
  • A higher-priority process arrives or becomes ready.
  • A scheduling policy determines that another process should run—for example, it has a shorter remaining time.

Examples: Round Robin (RR), Shortest Remaining Time First (SRTF), and preemptive Priority Scheduling.

Non-preemptive scheduling

In non-preemptive scheduling, once a process receives the CPU, the OS does not forcibly take it away. The process releases the CPU only when it:

  • terminates/completes its CPU burst, or
  • voluntarily blocks or moves to the waiting state, such as while requesting I/O.

Examples: First Come First Serve (FCFS), non-preemptive Shortest Job First (SJF), and non-preemptive Priority Scheduling.

Preemptive scheduling can interrupt P1 when P2 arrives; non-preemptive scheduling lets P1 retain the CPU until it finishes or blocks.

Once that process blocks and enters the Waiting state, it has voluntarily released the CPU, so the scheduler can run another ready process. “Non-preemptive” only means the OS does not forcibly take the CPU from a still-running process.
Memory analogy: In a non-preemptive meeting, the current speaker keeps the microphone until they finish or voluntarily pause. In a preemptive meeting, the moderator may stop the speaker when their time expires or an urgent speaker must be heard.

Chapter 2: Multiprogramming — Quick Revision

  • Definition: Multiple jobs are kept in main memory so another ready job can run when the current one waits.
  • Main goal: Maximize CPU utilization—“processor waits for no one.”
  • Job pool: Jobs on disk waiting for admission to main memory.
  • Single-CPU rule: Multiple jobs may be resident, but only one executes at an instant.
  • Switch trigger: Running job blocks for I/O/event or terminates; OS selects another ready job.
  • Advantages: Better utilization and throughput; CPU and I/O activities overlap.
  • Costs: Complex scheduling, memory management, protection, and context switching.
  • Time-sharing: Multiprogramming + frequent/preemptive CPU time slices + interactive response.
  • Illusion, not necessarily parallelism: Rapid context switching makes tasks appear simultaneous on one CPU/core.

Chapter 5: CPU Scheduling — Quick Revision

  • CPU scheduling: Selecting the next ready process to execute.
  • Preemptive: OS may force Running → Ready before the CPU burst completes.
  • Preemption triggers: time quantum expiry, higher-priority arrival, or scheduling-policy decision.
  • Preemptive examples: RR, SRTF, preemptive Priority.
  • Non-preemptive: Running process keeps CPU until it terminates or voluntarily blocks/waits.
  • Non-preemptive examples: FCFS, SJF, non-preemptive Priority.
  • Preemptive benefits/costs: Better responsiveness but more context-switch overhead, complexity, and possible starvation.
  • Non-preemptive benefits/costs: Simple and low overhead but poor response behind long jobs.
  • I/O trap: A blocked process has released the CPU; another ready process may run even in non-preemptive scheduling.
Fast test: Can the scheduler take the CPU from a process that is still running and has neither blocked nor finished? Yes = preemptive; No = non-preemptive.
