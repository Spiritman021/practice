Degree / Cardinality of Relationship

A relationship is the association between entities. The way entities are connected is called the degree or cardinality of the relationship. Common types are one-to-one, one-to-many, many-to-one, and many-to-many.

For many-to-many, write M:N instead of M:M to show both sides can have different counts.

One-to-One ER Example

In a one-to-one relationship, one entity instance is associated with at most one instance of another entity. Here, an Employee works in one Department, and a department is associated with one employee in this simplified example.

Relationship Table

The relationship table tells which employee works in which department. It must contain EID and DID. Both act as Foreign Keys referencing their parent tables.

In 1:1, neither EID nor DID should repeat inside the relationship table.
Primary key in relationship table: For one-to-one, either EID or DID can be selected as the Primary Key, because both are individually unique in the relationship table.
5

Can We Reduce Tables?

Yes. In a one-to-one relationship, the relationship table can be merged with either participating entity table. If EID is chosen as the Primary Key of Work, merge Work with Employee. Final number of tables becomes two.

Table Reduction in One-to-One Relationship
Before Reduction
Employee table + Department table + Work relationship table = 3 tables.

After Reduction
Merge Work into Employee or Department. Final result = 2 tables.

One-to-One Conversion Rules
Entities:Employee and Department become separate tables.
Relationship table:Work table contains EID and DID.
Foreign keys:EID references Employee(EID), DID references Department(DID).
Data rule:Neither EID nor DID repeats in Work for 1:1.
Primary key:Either EID or DID can become Primary Key of Work.
3
Table Reduction
Initial tables:Employee, Department, Work = 3 tables.
Can reduce?Yes, because relationship is one-to-one.
How?Merge Work with Employee if EID is selected, or with Department if DID is selected.
Final answer:2 tables.
