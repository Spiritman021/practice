BRIDGES IN COMPUTER NETWORKS

Connects two different LANs at L1+L2 — forwarding, filtering by MAC address, static vs dynamic (transparent) bridges, store-and-forward, and Spanning Tree Protocol. UGC NET & PSU focused.

1

Basic Identity & OSI Layer

A bridge is used to connect two different LANs. The key word is different — one side could be a Token Ring network (ring topology) and the other side could be Token Bus / Ethernet (bus topology). Bridges work at both the Physical Layer (L1) and the Data Link Layer (L2). Because they operate at L2, they can read and process MAC addresses — something repeaters and hubs cannot do.

2

Bridge Network Layout — P1 and P2

A bridge has two interfaces (ports): P1 on the left and P2 on the right. Devices M1–M4 connect to P1; devices M5–M8 connect to P2. All frames carry a source MAC and a destination MAC. The bridge inspects these addresses to decide whether to forward or filter.

🌐 Bridge Network Diagram — Two LANs Connected via P1 and P2
M1–M4 are on the P1 (left) side. M5–M8 are on the P2 (right) side. The bridge reads each frame's MAC address to decide: forward to the other side, or filter (block)?
3

Key Functionalities — Forwarding, Filtering & MAC Table

📤
Forwarding
If M1 sends a packet to M6, the bridge checks its MAC table and finds M6 is on the P2 side. Since the destination is on the opposite side, the bridge forwards the packet through to P2. The packet reaches M6.
✓ YES — Bridge CAN forward (destination on opposite side)
🚫
Filtering
If M1 sends a packet to M3, the bridge checks its MAC table and finds M3 is also on the P1 side. Since both source and destination are on the same side, the bridge filters (blocks) the packet — no need to flood the other segment with irrelevant traffic. This concept does NOT exist in repeaters or hubs.
✓ YES — Bridge CAN filter (destination on same side)
📈
MAC Address Table
The bridge maintains a MAC Address Table (similar to a routing table in a router) that maps each MAC address to the port (P1 or P2) it is reachable through. This table is the intelligence that enables forwarding and filtering decisions.

4

Types of Bridges — Static vs Dynamic (Transparent)

Bridges are classified by how they build their MAC Address Table. There are two types: Static (manual) and Dynamic / Transparent (self-learning).

✍️ Static Bridge

📌 The network administrator manually types each MAC address and its corresponding port into the table
📌 Table is fixed until an admin updates it — does not adapt on its own
📌 Problem 1: If a device's MAC address changes, the admin must manually update the entry
📌 Problem 2: If a device moves from one port to another (e.g., PC shifted to another lab), admin must update again
📌 No learning ability — error-prone; time-consuming
✕ Rarely used in real networks

🧠 Dynamic / Transparent Bridge

📌 Starts with a completely empty table
📌 Self-learning: Observes source MAC of every incoming frame and records which port it came from
📌 If destination unknown → broadcasts the frame to all ports; learns from the reply
📌 Continuously updates the table as traffic flows
📌 If a device moves, the table updates automatically the next time that device sends a frame
📌 Slower initially but self-correcting & zero admin effort
✓ Used in real networks — preferred
🧠 How a Dynamic Bridge Learns — Step by Step (M1 → M6, empty table)
1
M1 sends a frame to M6
The frame arrives at the bridge. Table is currently empty.
Frame: SRC=M1 (mac:01)  |  DST=M6 (mac:06)
2
Bridge learns M1 is on P1
Frame came in from Port P1, so the bridge records: "M1 is reachable via P1". Source learning.
Table entry added: M1 → P1
3
M6 is unknown → Broadcast
The bridge has no entry for M6, so it broadcasts the frame out of all other ports. M5, M6, M7, M8 all receive it.
Action: FLOOD to P2 (and all other ports)
4
M6 sends an acknowledgment back to M1
M6 replies. This reply frame arrives at the bridge from Port P2.
Frame: SRC=M6 (mac:06)  |  DST=M1 (mac:01)
5
Bridge learns M6 is on P2 & forwards to M1
Source of reply is M6, came from P2 → records "M6 is reachable via P2". Since M1 is already known to be on P1, the reply is forwarded correctly.
Table entry added: M6 → P2  |  Table now: M1→P1, M6→P2
6
Future frames: no more broadcast
Next time M1 sends to M6, the bridge already knows M6 is on P2 → direct forward, no broadcast. Over time the full table is built from normal traffic.
Result: M1→P1, M2→P1 … M6→P2, M7→P2 … (self-built table)
Dynamic bridges are slow at first (broadcast on every unknown destination) but get smarter over time. No administrator involvement needed.
⚠️ Static vs Dynamic at exam: Static = admin manually fills table; no learning. Dynamic = self-learning from source MAC of incoming frames; starts empty, floods unknown, builds table from replies. Dynamic is always preferred in real networks.
5

Store-and-Forward & Collision Domain

Unlike hubs and repeaters (which have no memory), a bridge uses the store-and-forward strategy. It stores an incoming frame in its buffer, processes it (checks MAC address), and only then forwards it. This prevents collisions.

💾
Store-and-Forward Strategy
When a frame arrives at the bridge, it is stored in the bridge's buffer (memory). If another frame arrives at the same time, it is also stored in the buffer — one after the other. There is no direct collision inside the bridge because packets are queued, processed, then forwarded.
✓ Bridge HAS buffer/memory — no simultaneous crash
💥
Collision Domain
In a hub, collision domain = N (all devices on the hub share one domain). In a bridge, each port is its own collision domain. The store-and-forward buffer prevents collisions at the bridge itself. Collisions are very rare in a bridged network compared to a hub or repeater.
✓ Bridge separates collision domains (unlike Hub/Repeater)

Spanning Tree Protocol (STP) — Removing Loops

When there are multiple paths between two network segments (e.g., for redundancy), a packet can get stuck in an infinite loop — travelling in circles between bridges forever. Routers solve this using the TTL (Time to Live) field which decrements on each hop and drops the packet when it hits zero. But bridges do not have a TTL field.

Instead, bridges use Spanning Tree Protocol (STP) to solve this. STP runs on all bridges in the network and together they construct a spanning tree — a loop-free logical topology. Any redundant (loop-causing) links are logically blocked. The spanning tree ensures there is exactly one active path between any two network points.

🔗 STP vs TTL:
• Router: Uses TTL field — decremented at each hop; packet dropped when TTL = 0
• Bridge: No TTL field → uses Spanning Tree Protocol to eliminate loops by blocking redundant paths

  • Frame arrives: read source MAC + port → add to table
