What is a "Key" in a Database?

In a database table, a key is a specific attribute (column) whose sole purpose is to uniquely identify any two tuples (rows). Every row in a table must be distinguishable from every other row — the key is what makes that possible.

🔑 Key = An attribute (or set of attributes) whose values are never repeated across different rows in a table. It acts as the unique fingerprint of each record.

Consider a Student table with only Name, City, and Age columns. Two students could share the same name, live in the same city, and be the same age. This creates a serious problem — especially in an online database shared by thousands of users.

Candidate Keys, Primary Keys & Alternate Keys

Once you identify all the uniquely-valued attributes in a table, a clear three-level hierarchy emerges. Understanding this hierarchy is essential for every database design task.

🗝️ Alternate Keys

Candidate Key Set → choose 1 as Primary Key → the rest become Alternate Keys. Simple hierarchy, critical concept.

Key Takeaway for Database Design

Choosing the right primary key is not random — it requires a deliberate design process. Here is the correct step-by-step approach every database designer should follow:

🛠️ Database Design Process — Keys
1
Identify all entities and their attributes
List every column you need for each table — Name, Age, City, Phone, Email, Aadhaar, etc.
2
Identify all naturally unique attributes → Candidate Key Set
From the full list, find every attribute whose values are never repeated across rows. Collect them all into the Candidate Key Set.
3
Choose the most suitable one → Primary Key
From the candidate set, strategically select one — consider which is most stable, most concise, and most meaningful. This becomes the Primary Key.
4
Remaining candidates → Alternate Keys
All other unique attributes stay in the table and remain available as Alternate Keys — useful for searches, lookups, and unique constraints.
Always establish the complete Candidate Key Set first — then strategically pick the Primary Key from it.
🎯 Key Takeaway: Primary Key ⊆ Candidate Key Set. Every primary key was once a candidate key. The candidate set is the foundation — identify it fully before choosing your primary key. This systematic approach is essential before moving on to E-R modeling and schema design.

What is a Key?
Definition:A specific attribute (column) whose values uniquely identify any two tuples (rows) in a table.
Core property:Key values are never repeated across different rows — each row gets a unique key value.
Problem it solves:Without a key, two rows with identical Name + City + Age are indistinguishable — is it a duplicate entry or two different people?
Example:Two rows: "Ravi Kumar, Delhi, 20" look identical. Add RollNo 1 vs RollNo 3 → instantly clear they are different students.
2
Unique Identifiers — Real-World Keys
For a Student:Aadhaar Number, Roll/Registration Number, Driving License, Voter ID, Phone Number, Email ID, PAN Card Number.
Why unique?No two different individuals can ever share the same value for any of these attributes.
General rule:Any attribute that is naturally unique across all records qualifies as a potential key.
3
Key Hierarchy — Candidate → Primary → Alternate
Candidate Key:The full set of all unique attributes — every attribute that can uniquely identify a row. All of Aadhaar, RollNo, Phone, Email, PAN etc. together form the Candidate Key Set.
Primary Key:The one candidate chosen as the official main identifier. Must be UNIQUE + NOT NULL. The most important key in database design.
Alternate Key:All remaining candidates after the Primary Key is selected. Still unique — just not the primary choice.
Formula:   Candidate Keys = Primary Key  +  Alternate Keys
Hierarchy:   Candidate Key Set  →  choose 1  →  Primary Key  |  rest  →  Alternate Keys
4
